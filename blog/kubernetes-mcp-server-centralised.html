One MCP Server, Eight Clusters: Centralised AI Access with IRSA

Richie Richard Rajkumar · April 2026 · 12 min read

In the previous article I covered what the Kubernetes MCP Server is and how it works. This article is about how to deploy it once for your entire team. Instead of every DevOps engineer installing a local binary and maintaining their own kubeconfig with eight cluster contexts, you deploy one MCP server in your centralised infrastructure cluster — and everyone connects to it with a single URL.

The result: same access, same guardrails, same cluster visibility for every engineer. No local kubeconfig distribution. No local MCP server installs. No configuration drift between team members. And with IRSA (IAM Roles for Service Accounts), the MCP server authenticates to all eight clusters using short-lived, automatically rotated credentials — no static tokens, no long-lived secrets mounted in pods.

The Problem with "Everyone Runs It Locally"

The default MCP server setup assumes each user runs the binary on their laptop. For a team of one, that's fine. For a DevOps team managing multiple clusters across environments, it creates real operational headaches:

Problem Local MCP Server Centralised MCP Server
Kubeconfig management Every engineer maintains 8 cluster contexts locally One kubeconfig, managed centrally
Credential rotation Static tokens per engineer, manual rotation IRSA — automatic, short-lived, no human touch
Guardrails consistency Each engineer configures their own TOML restrictions One config, enforced identically for everyone
Onboarding a new engineer Install binary, distribute kubeconfig, copy TOML config Add one URL to VS Code settings
Offboarding Hope they delete the kubeconfig and tokens Revoke VPN/network access — done
Audit trail Distributed across individual laptops Centralised logs, one place to query
Version consistency Engineers run different MCP server versions One version, one deployment

The question isn't whether centralising is better — it obviously is. The question is whether the MCP server supports it. It does.

The Cluster Landscape

Here's the setup we're working with — a real-world multi-cluster topology on EKS:

# 3 production clusters (application workloads)
apps-production-eu-central-1  # Production — Frankfurt
apps-production-us-east-1     # Production — N. Virginia
apps-production-ap-northeast-1# Production — Tokyo

# 3 sandbox clusters (testing, experiments)
apps-sandbox-eu-central-1     # Sandbox — Frankfurt
apps-sandbox-us-east-1        # Sandbox — N. Virginia
apps-sandbox-ap-northeast-1   # Sandbox — Tokyo

# 1 dev cluster (application development)
apps-development-eu-central-1 # Devlopment — Frankfurt

# 1 infra cluster (platform tooling)
infra-production-eu-central-1 # Grafana, GitHub Runners, ArgoCD, MCP Server

The infra cluster already has network connectivity to all seven application clusters — it runs Grafana dashboards that scrape metrics from them, ArgoCD that deploys to them, and GitHub runners that interact with them. The MCP server is just another platform tool that fits naturally into this cluster.

Why the infra cluster? It already has the network paths, the RBAC patterns, and the operational maturity. You're not establishing new trust boundaries — you're adding one more pod to a cluster that already talks to everything.

IRSA: Cross-Cluster Auth Without Static Credentials

IAM Roles for Service Accounts (IRSA) lets a Kubernetes ServiceAccount assume an AWS IAM role without any static credentials. EKS clusters expose an OIDC provider that AWS IAM trusts — when a pod runs with an annotated ServiceAccount, it gets a projected token that can be exchanged for temporary IAM credentials via STS.

For the MCP server, the chain works like this:

  1. The MCP server pod in infra-production-eu-central-1 runs with a ServiceAccount annotated with an IAM role
  2. IRSA injects a projected JWT token into the pod
  3. The pod uses that token to call sts:AssumeRole and get temporary credentials for the hub IAM role
  4. The hub role has trust policies that allow it to sts:AssumeRole into target roles in each of the 7 other AWS accounts
  5. Each target role is mapped to a Kubernetes RBAC ClusterRole in its respective cluster via aws-auth ConfigMap or EKS access entries
  6. The MCP server's kubeconfig uses the exec credential plugin with aws eks get-token to authenticate to each target cluster using the assumed role

No static tokens. No long-lived secrets. Credentials rotate automatically. If the pod restarts, it gets a fresh token. If IRSA annotation is removed, access dies instantly.

Step 1 — Create the Hub IAM Role in the Infra Account

This role lives in the AWS account that hosts infra-production-eu-central-1. It trusts the EKS OIDC provider so that the MCP server's ServiceAccount can assume it:

# Terraform — infra account (111111111111)

data "aws_eks_cluster" "infra" {
  name = "infra-production-eu-central-1"
}

module "mcp_server_irsa" {
  source  = "terraform-aws-modules/iam/aws//modules/iam-role-for-service-accounts-eks"
  version = "~> 5.0"

  role_name = "mcp-server-hub-role"

  oidc_providers = {
    main = {
      provider_arn = data.aws_eks_cluster.infra.identity[0].oidc[0].issuer
      namespace_service_accounts = ["platform-tools:mcp-server-sa"]
    }
  }
}

# Allow the hub role to assume target roles in other accounts
resource "aws_iam_role_policy" "assume_target_roles" {
  name = "assume-target-cluster-roles"
  role = module.mcp_server_irsa.iam_role_name

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect   = "Allow"
      Action   = "sts:AssumeRole"
      Resource = [
        # Production accounts
        "arn:aws:iam::222222222222:role/mcp-readonly-target",
        "arn:aws:iam::333333333333:role/mcp-readonly-target",
        "arn:aws:iam::444444444444:role/mcp-readonly-target",
        # Sandbox accounts
        "arn:aws:iam::555555555555:role/mcp-readonly-target",
        "arn:aws:iam::666666666666:role/mcp-readonly-target",
        "arn:aws:iam::777777777777:role/mcp-readonly-target",
        # Dev account
        "arn:aws:iam::888888888888:role/mcp-readonly-target",
      ]
    }]
  })
}

Step 2 — Create Target Roles in Each Cluster Account

Each AWS account that hosts a target cluster gets a mcp-readonly-target role that trusts the hub role. Apply this Terraform in each of the 7 target accounts:

# Terraform — each target account (e.g., 222222222222)

resource "aws_iam_role" "mcp_readonly_target" {
  name = "mcp-readonly-target"

  assume_role_policy = jsonencode({
    Version = "2012-10-17"
    Statement = [{
      Effect    = "Allow"
      Principal = { AWS = "arn:aws:iam::111111111111:role/mcp-server-hub-role" }
      Action    = "sts:AssumeRole"
    }]
  })
}

# Minimal permissions — only EKS cluster discovery + token generation
resource "aws_iam_role_policy" "eks_access" {
  name = "eks-describe-and-token"
  role = aws_iam_role.mcp_readonly_target.name

  policy = jsonencode({
    Version = "2012-10-17"
    Statement = [
      {
        Effect   = "Allow"
        Action   = ["eks:DescribeCluster", "eks:ListClusters"]
        Resource = "*"
      }
    ]
  })
}

Step 3 — Map Target Roles to Kubernetes RBAC

In each target cluster, map the mcp-readonly-target IAM role to a Kubernetes ClusterRole that grants read-only access. If you're using EKS access entries (the newer approach):

# Terraform — EKS access entry in each target cluster

resource "aws_eks_access_entry" "mcp_readonly" {
  cluster_name  = "apps-production-eu-central-1"
  principal_arn = "arn:aws:iam::222222222222:role/mcp-readonly-target"
  type          = "STANDARD"
}

resource "aws_eks_access_policy_association" "mcp_readonly" {
  cluster_name  = "apps-production-eu-central-1"
  principal_arn = "arn:aws:iam::222222222222:role/mcp-readonly-target"
  policy_arn    = "arn:aws:eks::aws:cluster-access-policy/AmazonEKSViewPolicy"

  access_scope {
    type = "cluster"
  }
}

Or if you're using the legacy aws-auth ConfigMap:

# aws-auth ConfigMap entry in each target cluster
mapRoles:
  - rolearn: arn:aws:iam::222222222222:role/mcp-readonly-target
    username: mcp-server
    groups:
      - mcp-readonly-group

For the aws-auth approach, also create a ClusterRoleBinding in each cluster:

apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: mcp-readonly
rules:
  - apiGroups: [""]
    resources: ["pods", "pods/log", "services", "events", "nodes", "configmaps", "namespaces", "persistentvolumeclaims"]
    verbs: ["get", "list", "watch"]
  - apiGroups: ["apps"]
    resources: ["deployments", "replicasets", "statefulsets", "daemonsets"]
    verbs: ["get", "list", "watch"]
  - apiGroups: ["batch"]
    resources: ["jobs", "cronjobs"]
    verbs: ["get", "list", "watch"]
  - apiGroups: ["networking.k8s.io"]
    resources: ["ingresses", "networkpolicies"]
    verbs: ["get", "list", "watch"]
---
apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRoleBinding
metadata:
  name: mcp-readonly-binding
roleRef:
  apiGroup: rbac.authorization.k8s.io
  kind: ClusterRole
  name: mcp-readonly
subjects:
  - kind: Group
    name: mcp-readonly-group
    apiGroup: rbac.authorization.k8s.io
Note the intentional omissions. The ClusterRole does not include secrets, serviceaccounts, roles, or rolebindings. Even if the MCP server's TOML config is misconfigured, the RBAC layer physically prevents access to sensitive resources. Defense in depth — the TOML config is the first gate, RBAC is the second.

Step 4 — Build the Multi-Cluster Kubeconfig

The kubeconfig mounted in the MCP server pod uses exec-based credential plugins to chain through the IRSA → hub role → target role path for each cluster:

apiVersion: v1
kind: Config

clusters:
  - name: apps-production-eu-central-1
    cluster:
      server: https://ABCDEF1234.gr7.eu-central-1.eks.amazonaws.com
      certificate-authority-data: LS0tLS1CRUd...
  - name: apps-production-us-east-1
    cluster:
      server: https://GHIJKL5678.gr7.us-east-1.eks.amazonaws.com
      certificate-authority-data: LS0tLS1CRUd...
  # ... repeat for all 8 clusters

users:
  - name: mcp-apps-production-eu-central-1
    user:
      exec:
        apiVersion: client.authentication.k8s.io/v1beta1
        command: aws
        args:
          - eks
          - get-token
          - --cluster-name
          - apps-production-eu-central-1
          - --role-arn
          - arn:aws:iam::222222222222:role/mcp-readonly-target
          - --region
          - eu-central-1
  - name: mcp-apps-production-us-east-1
    user:
      exec:
        apiVersion: client.authentication.k8s.io/v1beta1
        command: aws
        args:
          - eks
          - get-token
          - --cluster-name
          - apps-production-us-east-1
          - --role-arn
          - arn:aws:iam::333333333333:role/mcp-readonly-target
          - --region
          - us-east-1
  # ... repeat for each cluster with its account's target role ARN

contexts:
  - name: apps-production-eu-central-1
    context:
      cluster: apps-production-eu-central-1
      user: mcp-apps-production-eu-central-1
  - name: apps-production-us-east-1
    context:
      cluster: apps-production-us-east-1
      user: mcp-apps-production-us-east-1
  # ... repeat for all 8 clusters

current-context: apps-production-eu-central-1

Each user entry calls aws eks get-token with a specific --role-arn. Because the pod already has IRSA credentials (the hub role), the aws CLI automatically chains: IRSA token → hub role → AssumeRole → target role → EKS token. No static credentials anywhere in this file.

Deploying the MCP Server

The Kubernetes ServiceAccount

apiVersion: v1
kind: ServiceAccount
metadata:
  name: mcp-server-sa
  namespace: platform-tools
  annotations:
    eks.amazonaws.com/role-arn: arn:aws:iam::111111111111:role/mcp-server-hub-role

That annotation is the entire IRSA configuration. EKS sees it and projects a JWT token into any pod using this ServiceAccount.

The MCP Server Config

# mcp-config.toml

[server]
read_only = true

[restrictions]
blocked_resources = [
  "secrets",
  "serviceaccounts",
  "clusterroles",
  "clusterrolebindings",
  "roles",
  "rolebindings",
]

blocked_namespaces = [
  "kube-system",
  "kube-public",
  "kube-node-lease",
]

The Deployment

apiVersion: apps/v1
kind: Deployment
metadata:
  name: kubernetes-mcp-server
  namespace: platform-tools
  labels:
    app: kubernetes-mcp-server
spec:
  replicas: 2
  selector:
    matchLabels:
      app: kubernetes-mcp-server
  template:
    metadata:
      labels:
        app: kubernetes-mcp-server
    spec:
      serviceAccountName: mcp-server-sa
      containers:
        - name: mcp-server
          image: ghcr.io/containers/kubernetes-mcp-server:latest
          args:
            - "--port=8080"
            - "--stateless"
            - "--read-only"
            - "--kubeconfig=/etc/kubeconfig/config"
            - "--config=/etc/mcp/config.toml"
          ports:
            - containerPort: 8080
              name: http
          readinessProbe:
            httpGet:
              path: /stats
              port: 8080
            initialDelaySeconds: 5
            periodSeconds: 10
          livenessProbe:
            httpGet:
              path: /stats
              port: 8080
            initialDelaySeconds: 10
            periodSeconds: 30
          resources:
            requests:
              cpu: 100m
              memory: 128Mi
            limits:
              cpu: 500m
              memory: 256Mi
          volumeMounts:
            - name: kubeconfig
              mountPath: /etc/kubeconfig
              readOnly: true
            - name: mcp-config
              mountPath: /etc/mcp
              readOnly: true
      volumes:
        - name: kubeconfig
          secret:
            secretName: mcp-multi-cluster-kubeconfig
        - name: mcp-config
          configMap:
            name: mcp-server-config
---
apiVersion: v1
kind: Service
metadata:
  name: kubernetes-mcp-server
  namespace: platform-tools
spec:
  selector:
    app: kubernetes-mcp-server
  ports:
    - port: 80
      targetPort: 8080
      name: http

Key flags:

  • --port=8080 — enables HTTP/SSE transport (exposes /mcp and /sse endpoints)
  • --stateless — disables session stickiness so you can safely run 2+ replicas behind a Service
  • --read-only — blocks all mutations at the application layer (RBAC blocks them at the cluster layer too)

Expose via Ingress

apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: kubernetes-mcp-server
  namespace: platform-tools
  annotations:
    nginx.ingress.kubernetes.io/proxy-read-timeout: "3600"
    nginx.ingress.kubernetes.io/proxy-send-timeout: "3600"
    # SSE requires long-lived connections
    nginx.ingress.kubernetes.io/proxy-buffering: "off"
spec:
  ingressClassName: nginx
  tls:
    - hosts: [mcp.internal.company.com]
      secretName: mcp-tls
  rules:
    - host: mcp.internal.company.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: kubernetes-mcp-server
                port:
                  number: 80
Important: The Ingress annotations for proxy buffering and timeouts are critical. SSE connections are long-lived — if your ingress controller buffers responses or times out after 60 seconds, the connection will drop mid-conversation. Disable buffering and set generous timeouts.

What Every Engineer Adds: One URL

This is the entire client-side setup for each DevOps engineer. No binary install. No kubeconfig. No TOML config. Just this:

VS Code

// .vscode/mcp.json (or VS Code user settings)
{
  "servers": {
    "kubernetes": {
      "type": "sse",
      "url": "https://mcp.internal.company.com/sse"
    }
  }
}

Claude Desktop

{
  "mcpServers": {

      "type": "sse",

Claude Code (CLI)

That's it. Open VS Code, open Copilot or Claude, and ask: "Show me pods with more than 5 restarts across all production clusters." The MCP server handles context switching, authentication, and data aggregation. The engineer doesn't know or care about kubeconfigs, IRSA chains, or which account hosts which cluster.

What You Actually Get

Here's why this architecture is worth the setup effort:

Same Access for Everyone

Every engineer sees the same eight clusters with the same permissions. No one has a "special" kubeconfig with extra access because they were the one who set up the cluster. No one is locked out because they forgot to update their local context after a cluster migration. The MCP server is the single source of truth for cluster access.

Same Guardrails, Enforced Centrally

The TOML config that blocks Secrets and kube-system is applied once, in the ConfigMap. When you decide that networking.k8s.io/networkpolicies should also be blocked, you update one ConfigMap and restart the pods. Every engineer gets the new restriction immediately. No Slack message asking everyone to update their local config. No one running an outdated version with weaker restrictions.

Same Cluster Visibility

When an incident happens, everyone sees the same state. There's no "works on my machine" for cluster access — no stale cached tokens, no expired contexts, no mismatched cluster endpoints. If engineer A sees CrashLoopBackOff in apps-production-us-east-1, engineer B sees the same thing.

Zero Local Setup

Onboarding a new DevOps engineer takes one step: add the MCP server URL to their VS Code settings. They don't need aws configure, they don't need aws eks update-kubeconfig for eight clusters, they don't need to install the MCP binary, and they don't need a TOML config file. They just need VPN access to the internal domain.

Clean Offboarding

When someone leaves the team, you revoke their VPN or network access. No credentials to rotate, no tokens to invalidate, no kubeconfig files to chase down on their laptop. The MCP server's credentials (IRSA) are tied to the pod, not to any individual engineer.

Centralised Audit Trail

Every query and operation flows through one service. You can log every MCP request at the ingress layer, correlate it with the engineer's VPN identity, and feed it into your existing observability stack. Try doing that when each engineer runs a local binary on their laptop.

The Security Model: Three Independent Layers

This architecture gives you defense in depth with three independent security layers that work even if one of them fails:

Layer What It Does If It Fails
1. MCP Server Config TOML restrictions block resource types and namespaces; --read-only blocks all writes RBAC prevents the query from reaching the API server
2. Kubernetes RBAC ClusterRole limits verbs to get/list/watch and excludes sensitive resources MCP config already blocked the request before it was sent
3. IAM Permissions Target role only allows eks:DescribeCluster — no AWS resource access Even with IAM over-permission, RBAC restricts Kubernetes-level access

No single layer is sufficient. All three together mean that a misconfiguration in any one layer is caught by the other two.

Operational Tips from Running This in Production

  • Name your kubeconfig contexts clearly. Use apps-production-eu-central-1, not arn:aws:eks:eu-central-1:222222222222:cluster/apps-production-eu-central-1. The AI reads context names — clear names mean clear prompts. "Show me pods in apps-production-eu-central-1" is better than a 70-character ARN.
  • Pin the MCP server image tag. Don't use :latest in production. Pin to a specific release (e.g., :v0.0.58) and upgrade deliberately after testing in a staging deployment first.
  • Monitor the /stats endpoint. The MCP server exposes connection metrics. Wire it into Prometheus/Grafana to track active connections, request rates, and error rates.
  • The AWS CLI must be in the container image. The kubeconfig uses aws eks get-token as an exec credential plugin. If the MCP server container image doesn't include the aws CLI, token generation will fail. Build a custom image or use an init container to install it.
  • Set pod disruption budgets. With 2 replicas behind a Service, add a PDB with minAvailable: 1 so rolling updates don't cause full outages for your team.
  • Start with production read-only, sandbox read-write. Let engineers experiment with write operations in sandbox clusters while keeping production strictly read-only. When trust is established, consider selective writes (scale yes, delete no) for production.
My take: The centralised MCP server isn't just a convenience — it's a fundamentally better security model. With local MCP servers, you're distributing kubeconfigs with eight-cluster access to every engineer's laptop. With a centralised server, credentials never leave the cluster. Engineers interact through a single URL, controlled by your network policies and ingress rules. You get the same access, the same guardrails, and the same visibility for everyone — and you manage it in one place. That's not just easier. That's how platform tooling should work.
