From Helm to Kustomize: Why We Ditched Templates for Overlays

Richie Richard Rajkumar · May 2026 · 13 min read

We used Helm for everything. Third-party tools, internal applications, background workers, cron jobs — if it deployed to Kubernetes, it was a Helm chart. That's how most teams start. Helm is the default packaging tool for Kubernetes, and when you're setting up your first cluster, helm install is the fastest way to get things running.

Then we started maintaining our own charts. Go template logic crept in. values.yaml files grew to 200+ lines. Engineers spent more time debugging template rendering than writing actual Kubernetes manifests. The tool that was supposed to simplify deployments became the thing slowing them down. We moved our internal application deployments to Kustomize — and the complexity dropped overnight. Here's what drove the switch, what we gained, what we lost, and where Helm still earns its place.

What Helm Gets Right

Before the critique: Helm solves real problems, and it does some things better than any alternative.

  • Package management for Kubernetes. Helm charts package an entire application — Deployments, Services, ConfigMaps, RBAC, CRDs — into a versioned, distributable unit. helm install cert-manager installs 15+ resources with correct dependencies in one command. No other tool does this as well.
  • Third-party ecosystem. Every major Kubernetes project ships a Helm chart: AWS Load Balancer Controller, Prometheus, ArgoCD, KEDA, Karpenter, cert-manager. The chart authors handle the complexity of configuring their software correctly. You provide values; they handle the templates.
  • Release management. Helm tracks releases — what version is deployed, what values were used, when it was installed. helm rollback reverts to a previous release. helm history shows the audit trail. This state tracking is genuinely useful for operational visibility.
  • Lifecycle hooks. Pre-install jobs, post-upgrade migrations, pre-delete cleanup. Helm hooks let chart authors encode operational logic — run a database migration before the new deployment rolls out, clean up resources before deletion.
Helm is excellent at distributing other people's complexity. When you're consuming a chart someone else wrote, Helm's templating power is a feature — the chart author parameterised their application so you can configure it with values. The problem starts when you become the chart author for your own applications.

Where Helm Started Hurting

Go Templates Are Not YAML

This is the core issue. Helm templates look like YAML but aren't YAML. They're Go templates that produce YAML. Your editor can't validate them. Your linter can't parse them. A missing nindent produces valid YAML that does the wrong thing silently.

# This is not YAML. It's a Go template that renders YAML.
# Can you spot the bug?

spec:
  containers:
    - name: {{ .Values.app.name }}
      image: {{ .Values.image.repository }}:{{ .Values.image.tag }}
      resources:
        {{- toYaml .Values.resources | nindent 8 }}   # Should be nindent 8? Or 10?
      {{- if .Values.env }}
      env:
        {{- range .Values.env }}
        - name: {{ .name }}
          value: {{ .value | quote }}         # Forgot quote? Silent failure.
        {{- end }}
      {{- end }}

Every engineer on the team has been burned by nindent. The indentation in the rendered output depends on where the template directive sits in the file, not where the YAML should logically be indented. A miscount produces valid YAML with wrong structure — the pod spec gets a field at the wrong level, and you don't find out until kubectl apply fails or, worse, the pod starts with wrong configuration.

The values.yaml Indirection Problem

To understand what a Helm chart deploys, you need to hold two files in your head simultaneously: the template and the values. Neither makes sense alone. The template is full of {{ .Values.something }} placeholders. The values file is a flat list of keys with no indication of where they're used.

# values.yaml — what does this mean without reading the template?
replicaCount: 3
image:
  repository: my-registry/api
  tag: v1.4.2
resources:
  requests:
    cpu: "500m"
    memory: "512Mi"
ingress:
  enabled: true
  host: api.example.com
serviceAccount:
  create: true
  annotations: {}
autoscaling:
  enabled: false
  minReplicas: 2
  maxReplicas: 10

Now multiply this by four environments (dev, staging, production, canary), each with an override file. To know what production actually deploys, you need to mentally merge values.yaml + values-prod.yaml + any CLI --set overrides. That merge happens at render time, not in any file you can read.

Debugging Is Painful

When something goes wrong, your first step is helm template . --values values-prod.yaml to see the rendered output. But the rendered output is a single stream of YAML documents with no indication of which template file produced which resource. If the output has a problem, you trace it back through the template logic, the values, the helpers in _helpers.tpl, and any dependencies.

Compare this to Kustomize, where the output of kubectl kustomize overlays/prod is... the same YAML you wrote, with patches applied. The base is valid YAML. The overlay is valid YAML. The output is valid YAML. Every step is debuggable with standard YAML tools.

Chart Maintenance Overhead

Internal Helm charts accumulate complexity over time. Someone adds a conditional if .Values.monitoring.enabled. Someone else adds a range over sidecar containers. A third person adds helpers for label generation. Six months later, the chart has more template logic than actual Kubernetes configuration, and modifying it requires understanding the entire template dependency graph.

We had a chart for an internal API service that was 400 lines of Go templates to produce 120 lines of YAML. The ratio was inverted — more scaffolding than output.

Enter Kustomize

Kustomize takes the opposite approach: no templating. You write plain, valid Kubernetes YAML. For environment-specific differences, you write patches — small YAML fragments that modify specific fields. Kustomize merges them using strategic merge patches.

The Structure

# Directory structure for an app deployed to dev, staging, and prod
my-app/
  base/
    kustomization.yaml     # Lists resources to include
    deployment.yaml         # Full, valid, runnable deployment
    service.yaml            # Full, valid service
    hpa.yaml                # Full, valid HPA
  overlays/
    dev/
      kustomization.yaml   # References base + lists patches
      replicas-patch.yaml  # Only: replicas: 1
    staging/
      kustomization.yaml
      replicas-patch.yaml  # Only: replicas: 2
    prod/
      kustomization.yaml
      replicas-patch.yaml  # Only: replicas: 5
      resources-patch.yaml # Only: increased CPU/memory

The Base: Plain YAML

# base/deployment.yaml — this is real, valid, runnable YAML
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-server
spec:
  replicas: 3
  selector:
    matchLabels:
      app: api-server
  template:
    metadata:
      labels:
        app: api-server
    spec:
      containers:
        - name: api
          image: my-registry/api-server:v1.4.2
          ports:
            - containerPort: 8080
          resources:
            requests:
              cpu: "500m"
              memory: "512Mi"
            limits:
              memory: "1Gi"

This is a complete Deployment. You can kubectl apply -f base/deployment.yaml and it works. No rendering step. No values to substitute. No template to debug. Your editor validates it, your linter checks it, kubectl diff shows exactly what will change.

The Overlay: Only What Changes

# overlays/prod/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

resources:
  - ../../base

namespace: production

images:
  - name: my-registry/api-server
    newTag: v1.5.0

patches:
  - path: replicas-patch.yaml
  - path: resources-patch.yaml
# overlays/prod/replicas-patch.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-server
spec:
  replicas: 5
# overlays/prod/resources-patch.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-server
spec:
  template:
    spec:
      containers:
        - name: api
          resources:
            requests:
              cpu: "1"
              memory: "1Gi"
            limits:
              memory: "2Gi"

The production overlay says: take the base, set the namespace to production, update the image tag to v1.5.0, increase replicas to 5, and bump resource requests. That's it. Every file is valid YAML. The patches only contain what's different — not a copy of the entire manifest.

The key insight: With Helm, you start with templates and add values to make them concrete. With Kustomize, you start with concrete YAML and add patches to make it vary. Starting concrete is easier to understand, easier to debug, and easier to review in a pull request.

What Actually Made Us Switch

It wasn't one thing. It was the accumulation of friction across the team:

  1. Onboarding time. New engineers needed to learn Go templating, Helm's value resolution order, and our chart conventions before they could deploy their first service. With Kustomize, the base is just Kubernetes YAML — if you know kubectl, you know the base. Overlays are small patches. The learning curve dropped from days to hours.
  2. Pull request reviews. Reviewing a Helm values change required mentally rendering the template to understand the impact. Reviewing a Kustomize patch is reviewing YAML — the diff shows exactly what changes in the final output. kubectl kustomize overlays/prod | kubectl diff -f - shows the exact cluster impact before merging.
  3. GitOps compatibility. ArgoCD and Flux both support Kustomize natively. Our GitOps pipeline renders kustomize build and applies the result. With Helm, ArgoCD needs to run helm template at sync time, which means the rendered output isn't visible in Git — it's computed at deploy time. Kustomize output is deterministic and inspectable.
  4. The 80/20 rule. 80% of our internal charts used maybe 5% of Helm's capabilities — basic value substitution for image tags, replica counts, and resource limits. We were paying the complexity cost of a full templating engine to do what strategic merge patches handle natively.
  5. Drift between environments. With Helm, environment differences were scattered across multiple values-*.yaml files. With Kustomize, each overlay directory is self-contained — ls overlays/prod/ shows you every deviation from the base. No mental merging required.

Head-to-Head Comparison

Aspect Helm Kustomize
Approach Template + values → rendered YAML Base YAML + patches → merged YAML
Configuration files Go templates (not valid YAML) Plain Kubernetes YAML throughout
Environment variants Multiple values files, merged at render time Overlay directories with only the diffs
Learning curve Go templates + Helm conventions + value resolution Kubernetes YAML + patch mechanics
Debugging helm template → trace through templates kubectl kustomize → read the output
PR review Must mentally render to understand impact Diff shows the actual YAML change
Package distribution Chart repos, OCI registries, versioned releases No native packaging — just directories in Git
Release management Built-in: history, rollback, status tracking None — delegate to GitOps tool or kubectl
Lifecycle hooks Pre/post install, upgrade, delete hooks Not supported natively
Third-party software Dominant — every K8s project ships charts Growing — but Helm charts are still the standard

GitOps fit Supported but rendered at sync time Native — output is deterministic and in Git

Where We Still Use Helm

We didn't delete Helm. We stopped writing Helm charts. We still consume them for third-party software:

  • KEDA — helm install keda kedacore/keda
  • Karpenter — helm install karpenter oci://public.ecr.aws/karpenter/karpenter
  • cert-manager — complex CRD lifecycle that Helm handles well
  • AWS Load Balancer Controller — ALB/NLB provisioning, TargetGroupBinding CRDs, IAM integration
  • Prometheus / kube-prometheus-stack — hundreds of resources, dependencies between components
  • External Secrets Operator — CRDs + RBAC + webhook configuration

These are all cases where the chart author has encoded significant operational knowledge into the templates. Installing Prometheus from plain YAML would mean managing 50+ resources and understanding their dependencies. The Helm chart does that for you. This is Helm at its best — distributing other people's complexity.

The rule we follow: Consume Helm charts for software you didn't write. Use Kustomize for software you did write. Helm is a package manager; Kustomize is a deployment tool. They solve different problems.

The Hybrid Model That Works

In practice, most teams end up with a hybrid approach. Here's the model that works for us:

# Repository structure — hybrid Helm + Kustomize
infrastructure/
  helm-releases/             # Third-party charts managed by Helm
    karpenter/
      values.yaml             # Custom values for the Helm chart
    keda/
      values.yaml
    cert-manager/
      values.yaml
    aws-load-balancer-controller/
      values.yaml

applications/
  api-server/                # Internal app — Kustomize
    base/
    overlays/dev/
    overlays/staging/
    overlays/prod/
  worker-service/            # Internal app — Kustomize
    base/
    overlays/dev/
    overlays/staging/
    overlays/prod/

ArgoCD (or Flux) watches both directories. For helm-releases/, it uses the Helm source type. For applications/, it uses the Kustomize source type. Each tool is used where it's strongest. No compromise, no workaround.

Kustomize Built-In Features We Rely On

Kustomize isn't just patches. A few built-in features that replaced common Helm patterns for us:

# kustomization.yaml — built-in features
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization

resources:
  - ../../base

namespace: production               # Sets namespace on all resources

namePrefix: prod-                   # Prefixes all resource names

commonLabels:                        # Adds labels to all resources + selectors
  environment: production
  team: platform

images:                              # Image tag overrides without patching
  - name: my-registry/api-server
    newTag: v1.5.0

configMapGenerator:                  # Generates ConfigMaps with content hashing
  - name: app-config
    files:
      - config.yaml

secretGenerator:                     # Generates Secrets with content hashing
  - name: db-credentials
    envs:
      - .env.secret

The configMapGenerator and secretGenerator deserve special mention. They generate ConfigMaps and Secrets with a content hash suffix (e.g., app-config-m2k8f2h5). When the content changes, the name changes, which forces a Deployment rollout. With Helm, you typically need a checksum annotation hack to achieve the same thing. Kustomize does it by default.

How We Migrated

We didn't migrate everything at once. We followed a pragmatic approach:

  1. Stop writing new Helm charts. Any new service gets a Kustomize base + overlays from day one. This is the easiest step — no migration, just a new convention.
  2. Convert the simplest charts first. Services with minimal template logic — basically just value substitution for image tags and replicas. helm template the chart, clean up the output, and that becomes your Kustomize base. One afternoon per service.
  3. Tackle complex charts last. Charts with conditionals, loops, and helper functions. These take longer because you need to decide: does the conditional become a separate overlay, or does the feature just get included in the base? Usually the answer is "include it in the base, it's always on anyway."
  4. Don't migrate third-party charts. cert-manager, AWS Load Balancer Controller, Prometheus — these stay as Helm releases. The chart authors maintain them better than you could with raw YAML.
# Quick conversion: Helm chart → Kustomize base

# Step 1: Render the Helm chart to get the final YAML
$ helm template my-app ./charts/my-app \
    --values values-prod.yaml \
    --namespace production > rendered.yaml

# Step 2: Split into individual files
# (manually or with a tool like yq)
$ mkdir -p my-app/base
# Extract each resource into its own file

# Step 3: Create kustomization.yaml
$ cd my-app/base && kustomize create --autodetect

# Step 4: Create overlays for each environment
$ mkdir -p overlays/{dev,staging,prod}
# Add patches for environment-specific differences

# Step 5: Verify the output matches
$ kubectl kustomize overlays/prod | diff - rendered.yaml

Key Takeaways

  1. Helm is a package manager. Kustomize is a deployment tool. They solve different problems. Use Helm to install software you didn't write. Use Kustomize to deploy software you did write.
  2. Go templates are the tax you pay for Helm's power. For third-party charts, that tax is worth it — the chart author paid it, not you. For internal apps, the tax compounds: every engineer on the team pays it every time they touch the chart.
  3. Kustomize starts with valid YAML. The base is a complete, runnable deployment. Overlays only contain what changes per environment. Every file is valid YAML that your editor, linter, and kubectl understand natively.
  4. The hybrid model works. helm-releases/ for third-party software, applications/ with Kustomize for internal apps. ArgoCD or Flux handles both. Each tool is used where it's strongest.
  5. Pull request reviews get dramatically easier. Kustomize diffs show the actual YAML change. Helm diffs show values that need mental rendering to understand the impact.
  6. configMapGenerator and secretGenerator replace annotation hacks. Content-hash suffixed names force rollouts on config changes — no checksum annotation template required.
  7. Migration is incremental. Stop writing new charts first. Convert simple charts next. Tackle complex ones last. Don't touch third-party charts.
  8. The simplicity is the point. Kustomize doesn't have hooks, release history, or a chart repository. That's not a gap — it's a design choice. If you need those, use Helm. If you don't, why pay the complexity?
