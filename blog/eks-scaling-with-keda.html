KEDA on EKS: Event-Driven Scaling with SQS Queue Depth

Richie Richard Rajkumar · May 2026 · 12 min read

Your SQS worker pods are probably scaled with HPA based on CPU utilisation. That means they're always running — even when the queue is empty. You're paying for idle compute around the clock because HPA has no concept of "there's no work to do." It watches resource usage, not incoming demand. When the queue is empty but CPU sits at 30% from background processes, HPA happily keeps all your pods alive.

KEDA (Kubernetes Event-Driven Autoscaling) fixes this by reading the queue depth directly from SQS. Zero messages in the queue means zero worker pods. Fifty messages means five pods. It scales based on actual demand, not the side effects of demand. This guide walks through setting up KEDA on EKS with SQS, from installation through production hardening — including ScaledObject for persistent workers, ScaledJob for one-shot tasks, and the IAM configuration that makes it all work securely.

Why HPA Falls Short for Queue Workers

The Horizontal Pod Autoscaler scales pods based on observed resource metrics — CPU utilisation, memory consumption, or custom metrics you push into the metrics API. For request-serving workloads like web APIs, this works fine: more traffic means more CPU, which triggers more pods.

Queue workers are different. The relationship between resource usage and actual demand is indirect and delayed:

  • Empty queue, pods still running. HPA's minimum replica count keeps pods alive even when there's nothing to process. You can't set minReplicas: 0 — HPA doesn't support scale-to-zero.
  • Queue backlog building, CPU still low. If messages arrive faster than workers process them, the queue grows. But if each message is lightweight (an API call, a database update), CPU stays low and HPA doesn't react.
  • Slow response to spikes. HPA evaluates metrics on a fixed interval (default 15 seconds), then waits for a stabilisation window before scaling. By the time new pods are running, you've already missed your SLA.
  • No awareness of queue state. HPA doesn't know how many messages are waiting, how many are in-flight, or whether the Dead Letter Queue is growing. It only sees what the pod is doing, not what the pod should be doing.
Aspect HPA KEDA
Scaling signal CPU / Memory (resource usage) Queue depth (actual demand)
Scale to zero Not supported — minimum 1 replica Native — 0 pods when queue is empty
Response time 30–90 seconds (metric collection + stabilisation) Polling interval (default 30s, configurable to 5s)
Queue awareness None — blind to queue state Reads ApproximateMessageCount + in-flight
Event sources CPU, memory, custom metrics API 60+ scalers: SQS, Kafka, RabbitMQ, Prometheus, cron, etc.

What KEDA Actually Does

KEDA is a lightweight Kubernetes operator that extends the Kubernetes autoscaling primitives. It runs as a set of pods in your cluster (operator, metrics server, admission webhooks) and watches for two custom resources: ScaledObject and ScaledJob.

The core loop is simple:

  1. Poll the event source. KEDA queries SQS (or Kafka, RabbitMQ, Prometheus, etc.) at a configurable interval to get the current metric value — for SQS, that's ApproximateNumberOfMessages plus optionally ApproximateNumberOfMessagesNotVisible (in-flight messages).
  2. Compute desired replicas. KEDA divides the metric value by your configured threshold. If the queue has 50 messages and queueLength is 10, KEDA wants 5 pods.
  3. Scale the target. For a ScaledObject, KEDA manages an HPA under the hood (yes, it creates a real HPA object) and sets the desired replica count. For scale-to-zero specifically, KEDA handles that directly because HPA can't go below 1.
KEDA doesn't replace HPA — it wraps it. When replicas are between 1 and maxReplicaCount, KEDA delegates to a standard HPA with external metrics. The 0↔1 transition is handled by KEDA's own controller. This means KEDA plays nicely with existing HPA behaviours like scale-down stabilisation windows.

Installing KEDA on EKS

KEDA installs via Helm into its own namespace. Three pods: the operator, the metrics API server, and an admission webhook.

# Add the KEDA Helm repo and install
$ helm repo add kedacore https://kedacore.github.io/charts
$ helm repo update

$ helm install keda kedacore/keda \
    --namespace keda \
    --create-namespace \
    --version 2.16.1

Verify the installation:

$ kubectl get pods -n keda
NAME                                      READY   STATUS    AGE
keda-admission-webhooks-7f4d8b5c9-x2k4m   1/1     Running   45s
keda-operator-5c8f7d6b4-j9n2p             1/1     Running   45s
keda-operator-metrics-apiserver-6d9-r8t5   1/1     Running   45s

That's it. KEDA is now watching for ScaledObject and ScaledJob resources in all namespaces.

IAM and Authentication with EKS Pod Identity

KEDA needs to read from your SQS queue. On EKS, the cleanest way to grant this access is EKS Pod Identity (or IRSA if you're on an older setup). No long-lived credentials, no access keys mounted as secrets.

Step 1: Create the IAM Policy

KEDA only needs read access to query queue attributes — it never sends or deletes messages. Your worker pods handle that separately.

# IAM policy for KEDA's SQS scaler
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "sqs:GetQueueAttributes",
        "sqs:GetQueueUrl"
      ],
      "Resource": "arn:aws:sqs:eu-central-1:123456789012:order-processing-queue"
    }
  ]
}

Step 2: Create the IAM Role and Pod Identity Association

# Create the IAM role with EKS Pod Identity trust
$ aws iam create-role \
    --role-name keda-sqs-scaler-role \
    --assume-role-policy-document file://trust-policy.json

$ aws iam attach-role-policy \
    --role-name keda-sqs-scaler-role \
    --policy-arn arn:aws:iam::123456789012:policy/keda-sqs-read

# Associate with the KEDA operator's service account
$ aws eks create-pod-identity-association \
    --cluster-name production \
    --namespace keda \
    --service-account keda-operator \
    --role-arn arn:aws:iam::123456789012:role/keda-sqs-scaler-role

Step 3: Create the TriggerAuthentication

This tells KEDA how to authenticate when polling SQS. With Pod Identity, the config is minimal:

apiVersion: keda.sh/v1alpha1
kind: TriggerAuthentication
metadata:
  name: keda-aws-credentials
  namespace: default
spec:
  podIdentity:
    provider: aws
Why Pod Identity over mounted secrets? With Pod Identity (or IRSA), there are no credentials to rotate, no secrets to manage, and no risk of keys leaking into logs or error messages. The IAM role is assumed automatically via the EKS token exchange. If you're still using AWS_ACCESS_KEY_ID / AWS_SECRET_ACCESS_KEY in a Kubernetes Secret, this is your sign to migrate.

Configuring the ScaledObject

The ScaledObject is the core resource that connects your SQS queue to your worker Deployment. It tells KEDA: "watch this queue, and scale this Deployment based on the message count."

The Worker Deployment

First, your worker Deployment. Set replicas: 0 — KEDA manages the replica count from here.

apiVersion: apps/v1
kind: Deployment
metadata:
  name: sqs-worker
spec:
  replicas: 0          # KEDA controls this
  selector:
    matchLabels:
      app: sqs-worker
  template:
    metadata:
      labels:
        app: sqs-worker
    spec:
      serviceAccountName: sqs-worker-sa
      containers:
        - name: worker
          image: my-registry/sqs-worker:latest
          resources:
            requests:
              cpu: "250m"
              memory: "256Mi"
            limits:
              memory: "512Mi"
          env:
            - name: QUEUE_URL
              value: "https://sqs.eu-central-1.amazonaws.com/123456789012/order-processing-queue"

The ScaledObject

apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: sqs-worker-scaler
  namespace: default
spec:
  scaleTargetRef:
    name: sqs-worker             # Target Deployment
  minReplicaCount: 0               # Scale to zero when idle
  maxReplicaCount: 20              # Upper bound
  cooldownPeriod: 300              # Wait 5 min before scaling to 0
  pollingInterval: 30              # Check SQS every 30 seconds
  triggers:
    - type: aws-sqs-queue
      authenticationRef:
        name: keda-aws-credentials
      metadata:
        queueURL: "https://sqs.eu-central-1.amazonaws.com/123456789012/order-processing-queue"
        queueLength: "10"         # 1 pod per 10 messages
        awsRegion: "eu-central-1"
        scaleOnInFlight: "true"   # Count in-flight messages too

Here's what each field does:

  • queueLength: "10" — the threshold. KEDA divides the total message count by this value to compute desired replicas. 50 messages ÷ 10 = 5 pods.
  • scaleOnInFlight: "true" — includes messages currently being processed (visibility timeout) in the count. Without this, KEDA only counts messages waiting in the queue, which can cause under-scaling if processing is slow.
  • cooldownPeriod: 300 — after the last trigger activation, KEDA waits 5 minutes before scaling to zero. This prevents flapping if messages arrive in bursts.
  • pollingInterval: 30 — how often KEDA checks SQS. Lower values mean faster response but more API calls. 30 seconds is a good default; drop to 10–15 for latency-sensitive workloads.

ScaledJob for One-Shot Tasks

A ScaledObject scales a Deployment — long-lived pods that continuously poll the queue. But some workloads are better modelled as one-shot Jobs: video encoding, report generation, image processing, ETL pipelines. Each message triggers a Job, the Job processes it, and the Job completes.

ScaledJob creates Kubernetes Jobs based on queue depth instead of scaling a Deployment:

apiVersion: keda.sh/v1alpha1
kind: ScaledJob
metadata:
  name: video-encoder-job
spec:
  jobTargetRef:
    template:
      spec:
        serviceAccountName: encoder-sa
        containers:
          - name: encoder
            image: my-registry/video-encoder:latest
            resources:
              requests:
                cpu: "1"
                memory: "2Gi"
        restartPolicy: Never
    backoffLimit: 3
  pollingInterval: 15
  maxReplicaCount: 10
  successfulJobsHistoryLimit: 5
  failedJobsHistoryLimit: 5
  triggers:
    - type: aws-sqs-queue
      authenticationRef:
        name: keda-aws-credentials
      metadata:
        queueURL: "https://sqs.eu-central-1.amazonaws.com/123456789012/video-encode-queue"
        queueLength: "1"          # 1 Job per message
        awsRegion: "eu-central-1"
ScaledObject vs ScaledJob — when to use which: Use ScaledObject when your workers poll the queue continuously and process messages in a loop (most microservice workers). Use ScaledJob when each message represents a discrete unit of work that runs to completion — especially long-running tasks where you want Kubernetes Job semantics (retries, backoff, completion tracking).

Production Hardening

1. Don't Always Scale to Zero

Scale-to-zero is KEDA's headline feature, but it's not always what you want. When KEDA scales from 0 to 1, there's a cold-start delay: the pod needs to be scheduled, pulled (if not cached), started, and begin polling. Depending on your image size and node availability, this can be 15–45 seconds.

For latency-sensitive queues where messages need processing within seconds, set minReplicaCount: 1 to keep one warm pod running. The cost of one idle pod is almost always less than the cost of missed SLAs.

spec:
  minReplicaCount: 1    # Keep one warm pod for latency-sensitive queues
  maxReplicaCount: 20

2. Monitor KEDA Metrics

KEDA exposes Prometheus metrics that you should be watching:

# Key KEDA metrics for SQS scalers
keda_scaler_metrics_value           # Current queue depth as seen by KEDA
keda_scaler_errors_total            # Authentication or API failures
keda_scaled_object_is_active        # Whether the scaler is currently active (1) or idle (0)
keda_resource_registered_count      # Number of ScaledObjects/ScaledJobs being managed

Compare keda_scaler_metrics_value against the actual SQS ApproximateNumberOfMessages CloudWatch metric. If they diverge, you have an authentication issue or KEDA can't reach the queue.

3. Alert on Dead Letter Queue Growth

If messages keep failing and landing in the DLQ, your workers will process them, fail, and KEDA will keep scaling up to handle the growing queue — which generates more failures. This is a runaway loop.

Set a CloudWatch alarm on the DLQ's ApproximateNumberOfMessages. Any growth above zero should page someone. DLQ messages mean something is broken — either the message format changed, a downstream dependency is down, or your worker has a bug.

4. Set Sensible Cooldown and Polling Intervals

Parameter Default Recommendation
pollingInterval 30s 10–30s depending on latency requirements
cooldownPeriod 300s 300s for bursty queues, 60s for steady-state
queueLength 5 Match to your worker's throughput (messages/min)

The queueLength threshold is the most important tuning parameter. If your worker processes 10 messages per minute and you set queueLength: "10", each pod handles roughly one minute's worth of backlog. If you set it to "1", you get one pod per message — great for heavy tasks, wasteful for lightweight ones.

5. Pair KEDA with Karpenter for Node-Level Scaling

KEDA scales pods. But who scales the nodes those pods run on? Karpenter provisions right-sized EC2 instances in ~60 seconds when pods go Pending, and consolidates empty nodes when KEDA scales back to zero. Together, they create a fully event-driven pipeline: zero messages in the queue means zero pods and zero nodes — not just reduced compute, but no compute cost at all.

See Karpenter on EKS: KEDA + Karpenter Integration for the full setup with ScaledObject, Deployment, and NodePool manifests that work together.

Beyond SQS: Other Event Sources

SQS is one of 60+ event sources KEDA supports. The same pattern applies to any async workload:

Event Source Scaler Type Scaling Metric
Apache Kafka apache-kafka Consumer group lag (unprocessed messages)
RabbitMQ rabbitmq Queue length or message rate
Amazon Kinesis aws-kinesis-stream Shard iterator age (how far behind)
Prometheus prometheus Any PromQL query result
Cron cron Time-based schedule (scale up at 9am, down at 6pm)
Azure Service Bus azure-servicebus Queue / topic message count
Google Cloud Pub/Sub gcp-pubsub Subscription message count

You can even combine multiple triggers on a single ScaledObject. For example, scale a deployment based on both SQS queue depth and a Prometheus metric for request latency — KEDA uses the trigger that produces the highest replica count.

The savings scale linearly with the number of queue-based services you run. If you have 10 worker Deployments all running idle overnight, the numbers add up fast.

Key Takeaways

  1. HPA can't scale queue workers effectively — it watches CPU/memory, not queue depth. Workers stay running even when there's nothing to process.
  2. KEDA scales pods based on actual demand — it reads ApproximateNumberOfMessages directly from SQS and computes the exact number of pods needed.
  3. Scale-to-zero eliminates idle compute — when the queue is empty, your worker pods don't exist. No pods, no cost.
  4. Use EKS Pod Identity for authentication — no long-lived credentials, no secrets to rotate. KEDA's TriggerAuthentication with podIdentity: aws is the cleanest setup.
  5. ScaledObject for persistent workers, ScaledJob for one-shot tasks — match the KEDA resource to your workload pattern.
  6. Don't blindly scale to zero — latency-sensitive queues should keep minReplicaCount: 1 to avoid cold starts.
  7. Pair KEDA with Karpenter — KEDA scales pods, Karpenter scales nodes. Together they create a fully event-driven pipeline where you only pay for compute when there's actual work.
  8. Monitor and alert — watch keda_scaler_metrics_value for drift and alarm on DLQ growth to prevent runaway scaling loops.
