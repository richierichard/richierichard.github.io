S3 Files: Your Buckets Are File Systems Now

Richie Richard Rajkumar · April 2026 · 10 min read

For as long as S3 has existed, there's been a tradeoff: you get cheap, durable, infinitely scalable object storage — but you can't cd into it. You can't vim a config file. You can't mount it and let your application read from a path. If your workload needed a file system, you reached for EFS or FSx, copied the data over, and built sync logic to keep them in sync. That era just ended.

Amazon S3 Files, launched in April 2026, turns any general purpose S3 bucket into a mountable NFS v4.1+ file system. You mount it on EC2, EKS pods, ECS containers, or Lambda functions — and your S3 objects appear as files and directories. Writes to the file system sync back to S3 automatically. Changes to S3 objects appear in the mount within seconds. One bucket, two access patterns, no data movement.

What Actually Changed

S3 Files adds a file system layer on top of any general purpose S3 bucket. Under the hood, it uses EFS infrastructure to provide NFS access with ~1ms latencies for active data. The key mechanics:

  • NFS v4.1+ protocol — standard POSIX file operations: create, read, update, delete, rename, chmod, chown
  • Bidirectional sync — writes on the file system export back to S3 within minutes; changes on S3 appear in the file system within seconds
  • High-performance storage tier — frequently accessed files and metadata are cached on low-latency storage; cold files are served directly from S3
  • Byte-range reads — only the requested bytes are transferred from S3, not the entire object
  • Multi-attach — multiple compute resources can mount the same file system concurrently with NFS close-to-open consistency
  • Available in all commercial AWS regions — no waitlist, no preview
Aspect Before S3 Files With S3 Files
Reading S3 data from an app SDK calls, download to local disk, or S3 FUSE mount (slow, limited) cat /mnt/s3/config/app.yaml
Editing a file in S3 Download → edit → re-upload entire object echo "new_value" >> /mnt/s3/config/app.yaml
Sharing data across compute Copy to EFS, build sync scripts, manage two storage systems Multiple EC2/EKS/Lambda mount the same S3 file system
ML training data pipeline S3 → copy to FSx for Lustre → train → copy results back to S3 Mount S3 bucket directly, train, results appear in S3 automatically
Latency for active data S3 GET: ~10-50ms first byte ~1ms from high-performance storage tier
POSIX permissions Not applicable — S3 uses IAM policies UID/GID-based permissions stored as object metadata
This is not S3 Mountpoint. Mountpoint for S3 is an open-source FUSE client that provides read-heavy, sequential access to S3. S3 Files is a fully managed NFS file system with write support, sub-millisecond latency, concurrent multi-attach, and automatic bidirectional sync. They solve different problems.

How It Works: Three Steps

Step 1 — Create the S3 File System

You can create a file system from the S3 console, CLI, or IaC. The file system is attached to an existing general purpose bucket:

# Create the file system
$ aws s3 create-file-system \
    --bucket-name ml-training-data-prod

# Output:
# {
#   "FileSystemId": "fs-0aa860d05df9afdfe",
#   "BucketName": "ml-training-data-prod",
#   "Status": "CREATING"
# }

Step 2 — Create a Mount Target

A mount target is a network endpoint in your VPC — it's how your compute resources reach the file system. You need one per availability zone:

# Create mount target in your subnet
$ aws s3 create-mount-target \
    --file-system-id fs-0aa860d05df9afdfe \
    --subnet-id subnet-0a1b2c3d4e5f67890 \
    --security-groups sg-0abcdef1234567890

# The security group must allow NFS traffic (TCP 2049) from your compute resources

Step 3 — Mount and Use

# Install or update the EFS utils (required for the s3files mount type)
$ sudo yum install -y amazon-efs-utils

# Create the mount point and mount
$ sudo mkdir -p /mnt/s3
$ sudo mount -t s3files fs-0aa860d05df9afdfe:/ /mnt/s3

# That's it. Your S3 objects are now files.
$ ls /mnt/s3/
config/  data/  logs/  assets/

$ cat /mnt/s3/config/app.yaml
database:
  host: prod-db.internal
  port: 5432

# Write a file — it syncs to S3 automatically within minutes
$ echo "deployment: v2.14.0" > /mnt/s3/config/deploy.yaml

# Verify it's in S3
$ aws s3 ls s3://ml-training-data-prod/config/deploy.yaml
2026-04-09 14:32:07         22 deploy.yaml

Mounting on EKS: PersistentVolume with the EFS CSI Driver

For Kubernetes workloads, you mount S3 Files using the existing EFS CSI driver — because S3 Files uses EFS infrastructure under the hood. If you're already running EFS volumes in EKS, you already have the driver installed.

Static Provisioning with a PersistentVolume

apiVersion: v1
kind: PersistentVolume
metadata:
  name: s3-files-training-data
spec:
  capacity:
    storage: 100Gi  # Required field but not enforced — S3 is effectively unlimited
  accessModes:
    - ReadWriteMany  # Multiple pods can mount concurrently
  persistentVolumeReclaimPolicy: Retain
  csi:
    driver: efs.csi.aws.com
    volumeHandle: fs-0aa860d05df9afdfe  # Your S3 file system ID
---
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: training-data-claim
  namespace: ml-pipelines
spec:
  accessModes:
    - ReadWriteMany
  resources:
    requests:
      storage: 100Gi
  storageClassName: ""
  volumeName: s3-files-training-data

Using It in a Pod

apiVersion: v1
kind: Pod
metadata:
  name: training-job
  namespace: ml-pipelines
spec:
  containers:
    - name: trainer
      image: ml-team/model-trainer:v3.2
      command: ["python", "train.py", "--data-dir=/data"]
      volumeMounts:
        - name: training-data
          mountPath: /data
  volumes:
    - name: training-data
      persistentVolumeClaim:
        claimName: training-data-claim

The training job reads from /data as if it were a local directory. The data lives in S3, served through the S3 Files layer. When the job writes checkpoints or results back to /data/output/, those files appear in the S3 bucket automatically.

ReadWriteMany is the key. Multiple pods across different nodes can mount the same S3 file system concurrently. This is what makes it viable for distributed ML training, shared config management, and multi-pod pipelines — without the EBS single-attach limitation or the need to run a separate EFS file system.

Terraform: Full Example

resource "aws_s3_file_system" "training_data" {
  bucket_name = "ml-training-data-prod"

  tags = {
    Environment = "production"
    Team        = "ml-platform"
  }
}

resource "aws_s3_mount_target" "az_a" {
  file_system_id  = aws_s3_file_system.training_data.id
  subnet_id       = "subnet-0a1b2c3d4e5f67890"  # AZ-a subnet
  security_groups = [aws_security_group.s3_files_nfs.id]
}

resource "aws_s3_mount_target" "az_b" {
  file_system_id  = aws_s3_file_system.training_data.id
  subnet_id       = "subnet-0b2c3d4e5f6789012"  # AZ-b subnet
  security_groups = [aws_security_group.s3_files_nfs.id]
}

resource "aws_security_group" "s3_files_nfs" {
  name_prefix = "s3-files-nfs-"
  vpc_id      = var.vpc_id

  ingress {
    from_port       = 2049
    to_port         = 2049
    protocol        = "tcp"
    security_groups = [var.compute_security_group_id]  # Allow from EC2/EKS nodes
  }
}

output "file_system_id" {
  value = aws_s3_file_system.training_data.id
}

Lambda: Mount S3 as a File System in a Function

Lambda functions can mount S3 Files just like they can mount EFS today. This is particularly powerful for functions that process files — instead of downloading from S3 into /tmp (capped at 10 GB), the function reads directly from the mount:

# Terraform — Lambda with S3 Files mount

resource "aws_lambda_function" "processor" {
  function_name = "data-processor"
  role          = aws_iam_role.lambda_role.arn
  handler       = "index.handler"
  runtime       = "python3.12"
  timeout       = 300

  vpc_config {
    subnet_ids         = var.private_subnet_ids
    security_group_ids = [aws_security_group.lambda_sg.id]
  }

  file_system_config {
    arn              = aws_s3_file_system.training_data.arn
    local_mount_path = "/mnt/data"
  }
}
# Lambda function code — reads directly from the mounted file system
import json
import os

def handler(event, context):
    # No S3 SDK calls needed — just read from the mount
    config_path = "/mnt/data/config/app.yaml"

    with open(config_path, "r") as f:
        config = f.read()

    # Process files in a directory — standard Python, no boto3
    data_files = os.listdir("/mnt/data/incoming/")

    for filename in data_files:
        filepath = os.path.join("/mnt/data/incoming", filename)
        with open(filepath, "r") as f:
            data = json.load(f)
            # process data...

        # Move processed files — standard file operations
        os.rename(filepath, filepath.replace("/incoming/", "/processed/"))

    return {"processed": len(data_files)}

No boto3, no s3.get_object(), no streaming response bodies, no temp files. Just open() and read(). The Lambda function reads and writes to S3 through standard file I/O.

Where This Changes the Architecture

1. ML Training Pipelines

The classic pain: data lives in S3, but training frameworks (PyTorch, TensorFlow) expect a file path. Previously you'd copy to FSx for Lustre, train, then copy results back. With S3 Files, mount the bucket directly. The training job reads data from the mount at ~1ms latency. Checkpoints and model outputs write back to S3 automatically. No copy step, no FSx management, no sync scripts.

2. Shared Config Across a Cluster

Config files, feature flags, model weights — things that multiple pods or instances need to read. Instead of baking them into images or using a sidecar that polls S3, mount the bucket. Update the file in S3, it appears in the mount within seconds across all nodes.

3. Log Processing and ETL

Lambda functions that process incoming files no longer need to download from S3 to /tmp. Mount the bucket, read from /mnt/data/incoming/, write processed results to /mnt/data/output/. The files move from one S3 prefix to another — through standard file operations, not SDK calls.

4. AI Agents and Tool-Using LLMs

AI agents that interact with file-based tools (Python scripts, shell commands, code interpreters) can now access S3 data through a file system interface. The agent doesn't need S3-specific tooling — it uses standard file operations. Mount the bucket into the agent's sandbox and it can read, write, and organise files naturally.

What to Watch Out For

  • Sync is not instant. File system → S3 sync takes minutes, not milliseconds. If your workflow depends on a file appearing in S3 immediately after writing, you'll need to account for this delay. S3 → file system sync is faster (seconds) but still not real-time.
  • Close-to-open consistency. Changes are visible to other clients after the file is closed. If pod A writes to a file and pod B reads it, pod B sees the update only after pod A closes the file. This is standard NFS semantics, but it matters for concurrent workloads.
  • General purpose buckets only. S3 Files works with general purpose buckets — not directory buckets (S3 Express One Zone). If your data is in Express One Zone for latency reasons, you can't use S3 Files on it.
  • Cost is additive. You pay for S3 storage + S3 Files high-performance storage + S3 requests during sync + NFS I/O. For read-heavy, sequential workloads where latency doesn't matter, plain S3 with Mountpoint may still be cheaper.
  • Security groups matter. The mount target needs a security group that allows TCP 2049 (NFS) from your compute resources. If your EKS nodes are in a different security group than your EC2 instances, both groups need ingress rules.
  • EFS utils required. The amazon-efs-utils package must be installed on your instances or baked into your AMI. AWS-provided AMIs have it preinstalled, but custom images may not.

S3 Files vs EFS vs FSx vs Mountpoint: When to Use What

Service Best For Protocol Latency
S3 Files Data that lives in S3 and needs file system access — ML, shared config, Lambda processing NFS v4.1+ ~1ms (active)
EFS File-native workloads that don't need S3 as the source of truth — CMS, web content, home directories NFS v4.1 ~0.5ms
FSx for Lustre HPC, GPU clusters, massive parallel I/O (linked to S3 but purpose-built for throughput) Lustre sub-ms
Mountpoint for S3 Read-heavy, sequential access to S3 — data lakes, analytics, large file reads (no write support) FUSE ~10-50ms
The decision tree is simple: If your data is in S3 and your compute needs a file path to access it with low latency and write support → S3 Files. If your data is file-native and never needs to be in S3 → EFS. If you need extreme parallel throughput for HPC → FSx for Lustre. If you just need cheap, read-only sequential access to S3 → Mountpoint.

Key Takeaways

  1. S3 Files makes any general purpose bucket mountable as an NFS v4.1+ file system — from EC2, EKS, ECS, and Lambda
  2. Bidirectional sync is automatic — writes on the file system appear in S3 within minutes, S3 changes appear in the file system within seconds
  3. ~1ms latency for active data via a high-performance storage tier, with cold data served directly from S3
  4. Multi-attach with ReadWriteMany — multiple pods, instances, or functions can mount the same file system concurrently
  5. Uses the EFS CSI driver in EKS — if you already run EFS volumes, the driver is already installed
  6. This is not Mountpoint for S3 — Mountpoint is read-heavy FUSE; S3 Files is a full read/write NFS file system with sub-millisecond latency
  7. Available in all commercial regions today — no preview, no waitlist
