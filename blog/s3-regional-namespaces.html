AWS S3 Bucket Naming Just Changed Forever

Richie Richard Rajkumar · March 2026 · 6 min read

If you manage AWS infrastructure at any meaningful scale, you've probably felt the friction of S3's global bucket namespace. Names like my-company-prod-data-eu-west-1-2024 exist not because engineers love verbose names — they exist because someone already claimed data, assets, backups, and every other sane option years ago. That changed in March 2026.

AWS introduced account-regional namespaces for S3 general purpose buckets, scoping bucket names to a specific account and region. The mechanism is a structured naming convention — {prefix}-{accountId}-{region}-an — that encodes the namespace directly into the bucket name. Here's what actually changed, what the convention means, and what you need to update in your IaC.

Why the Global Namespace Was Always a Problem

S3 launched in 2006 with a single global namespace. Every bucket name across every AWS account in the world had to be unique. At the time, the S3 user base was small enough that this worked. By 2026, it had become a genuine operational problem:

  • Bucket squatting — attackers registering predictable bucket names (e.g. company-logs, company-backups) before target organisations do. A deleted bucket's name immediately becomes available to anyone globally.
  • Name exhaustion — short, meaningful names are almost universally taken across the entire AWS customer base
  • Naming anti-patterns — teams were forced to append account IDs or timestamps to every bucket name just to guarantee uniqueness, creating exactly the pattern the new convention now formalises
  • Multi-account drift — maintaining consistent naming conventions across dev, staging, and prod accounts was impossible when names had to be globally unique
  • No availability guarantee — even if you deleted a bucket, another account could immediately claim that name before you recreated it
The core shift: Bucket names are now scoped to your AWS account and region. Two different accounts can use the same logical prefix in the same region — each gets their own isolated namespace, and the bucket name itself encodes the account and region explicitly.

What Actually Changed

From March 2026, buckets created with the new account-regional namespace use a structured naming convention that encodes the account and region directly into the bucket name. The ARN format is unchanged — the uniqueness is achieved through naming, not a new ARN scheme.

Aspect Old (Global) New (Account-Regional)
Uniqueness scope Global — all accounts worldwide Account + region only
Bucket name format my-bucket (bare, manually unique) my-bucket-{accountId}-{region}-an
ARN format arn:aws:s3:::bucket-name arn:aws:s3:::bucket-name — unchanged
Name availability guarantee None — anyone can claim deleted names Guaranteed — scoped to your account
Multi-account consistency Impossible — names must differ per account Same prefix works across all your accounts
CreateBucket API No extra params Add --bucket-namespace account-regional

The New Naming Convention

This is the central mechanic. Account-regional buckets follow a structured format that makes the namespace explicit in the name itself:

{prefix}-{AWS-Account-ID}-{AWS-Region}-an

# Real examples
app-logs-111122223333-eu-west-1-an
tf-state-999988887777-us-east-1-an
assets-012345678910-ap-southeast-1-an

Breaking down the components:

  • {prefix} — your logical bucket name, up to 37 characters (leaving room for the suffix within the 63-char total limit)
  • {AWS-Account-ID} — your 12-digit account ID, embedded directly in the name
  • {AWS-Region} — the full region code where the bucket lives
  • -an — the account namespace suffix, required to identify this as an account-regional bucket
Why this format? By encoding account ID and region into the name, the bucket is globally identifiable without requiring a global uniqueness check. The -an suffix signals to AWS (and to your team) that this bucket lives in an account-scoped namespace rather than the old global pool.

Two different accounts can both create a bucket with the same prefix — the resulting names are different because the account ID differs. Your dev, staging, and prod accounts can all have an app-logs bucket with predictable, consistent naming:

# Dev account (111122223333)
app-logs-111122223333-eu-west-1-an

# Staging account (444455556666)
app-logs-444455556666-eu-west-1-an

# Prod account (999988887777)
app-logs-999988887777-eu-west-1-an

What You Need to Update

AWS CLI

Pass --bucket-namespace account-regional when creating buckets. The --bucket value should be the full structured name:

aws s3api create-bucket \
    --bucket app-logs-111122223333-eu-west-1-an \
    --bucket-namespace account-regional \
    --region eu-west-1 \
    --create-bucket-configuration LocationConstraint=eu-west-1

CloudFormation

Two options depending on whether you want to specify the full name or let CloudFormation build it:

# Option 1 — explicit full name (using pseudo-parameters)
Type: AWS::S3::Bucket
Properties:
  BucketName: !Sub "app-logs-${AWS::AccountId}-${AWS::Region}-an"
  BucketNamespace: account-regional

# Option 2 — prefix only, CloudFormation appends the suffix
Type: AWS::S3::Bucket
Properties:
  BucketNamePrefix: app-logs
  BucketNamespace: account-regional
# → resolves to: app-logs-{accountId}-{region}-an

Terraform

Use the data.aws_caller_identity and data.aws_region data sources to build the name dynamically. This way the same module works consistently across all your accounts:

data "aws_caller_identity" "current" {}
data "aws_region" "current" {}

resource "aws_s3_bucket" "app_logs" {
  bucket = "app-logs-${data.aws_caller_identity.current.account_id}-${data.aws_region.current.name}-an"
}

# Example output: app-logs-111122223333-eu-west-1-an

Boto3 (Python)

Pass BucketNamespace in the CreateBucketConfiguration parameter:

# boto3 — create an account-regional bucket
import boto3

s3 = boto3.client('s3', region_name='eu-west-1')

s3.create_bucket(
    Bucket='app-logs-111122223333-eu-west-1-an',
    CreateBucketConfiguration={
        'LocationConstraint': 'eu-west-1',
        'BucketNamespace': 'account-regional'
    }
)

Enforcing It via SCP

Lock down your organisation so only account-regional buckets can be created — no one accidentally creates old-style global buckets:

{
  "Sid": "RequireAccountRegionalBuckets",
  "Effect": "Deny",
  "Action": "s3:CreateBucket",
  "Resource": "*",
  "Condition": {
    "StringNotEquals": {
      "s3:x-amz-bucket-namespace": "account-regional"
    }
  }
}

What You Don't Need to Worry About

A few things that are not affected by this change:

  • ARN format — remains arn:aws:s3:::bucket-name. No changes to IAM policies referencing bucket ARNs
  • Existing buckets — they stay on the global namespace; migration is opt-in
  • Performance and cost — zero impact, this is purely a naming and namespace change
  • S3 APIs — all existing API operations (GetObject, PutObject, ListBuckets, etc.) work unchanged
  • Access patterns — presigned URLs, static website hosting, event notifications, replication, and lifecycle rules are unaffected
  • AWS service integrations — CloudFront, Lambda, Athena, Glue, and all other services that interact with S3 work without modification
My take: For 18 years, every DevOps engineer has cargo-culted account IDs and region codes into bucket names manually — precisely because the global namespace forced it. AWS has now formalised that pattern, added the -an suffix, and made the scoping enforceable. The upgrade path is straightforward: adopt the convention for new buckets, use CloudFormation's BucketNamePrefix to generate names automatically, and add an SCP to keep your org clean. Existing buckets need nothing — which means you can move at your own pace.

Key Takeaways

  1. New account-regional buckets follow the format {prefix}-{accountId}-{region}-an — the naming convention makes the scope explicit
  2. The -an suffix is mandatory and identifies the bucket as account-regional, not global
  3. ARN format is unchanged — arn:aws:s3:::bucket-name still works; no IAM policy rewrites needed
  4. All creation paths require opting in: pass --bucket-namespace account-regional in the CLI, BucketNamespace: account-regional in CloudFormation, or BucketNamespace in SDK calls
  5. Use s3:x-amz-bucket-namespace in SCPs to enforce account-regional creation org-wide
  6. Existing global-namespace buckets are unaffected — migration is opt-in, no forced cutover
