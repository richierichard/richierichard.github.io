Cluster Autoscaler vs Karpenter: Choosing the Right Node Scaler for EKS

Richie Richard Rajkumar · May 2026 · 13 min read

Both Cluster Autoscaler and Karpenter solve the same fundamental problem: when pods can't schedule because there's no node with enough capacity, provision a new one. When nodes sit idle, remove them. But they approach this problem from opposite directions. Cluster Autoscaler asks "which of my pre-defined node groups should I scale?" Karpenter asks "what does this specific pod actually need?"

That architectural difference ripples through everything — provisioning speed, instance selection, cost efficiency, operational overhead, and how the system reacts when things change. This article is a practical decision framework: when each autoscaler is the right choice, where each one falls short, and how to migrate if you decide to switch.

The Architectural Difference

Before comparing features, you need to understand why these two tools behave differently. It comes down to one design decision: what sits between the pending pod and the EC2 instance?

Cluster Autoscaler: The ASG Intermediary

Cluster Autoscaler (CAS) manages nodes indirectly through Auto Scaling Groups. You define ASGs ahead of time — each with a launch template specifying instance types, an AMI, networking config, and min/max bounds. When pods go Pending, CAS evaluates which ASG can accommodate them and increases that group's desired capacity. The ASG then launches an instance using its launch template.

This means every scaling decision is filtered through the ASG abstraction. CAS doesn't pick the instance type — the ASG does. CAS doesn't know about AMIs — the launch template does. CAS doesn't handle interruptions — a separate DaemonSet (Node Termination Handler) does. CAS orchestrates ASGs; the ASGs manage the actual infrastructure.

Karpenter: Direct Provisioning

Karpenter removes the ASG intermediary entirely. When pods go Pending, Karpenter evaluates their resource requirements, batches compatible pods, and calls the EC2 Fleet API directly to launch the cheapest instance that fits. No ASG, no launch template, no intermediary. The instance type, AZ, and purchase option are chosen at the moment of provisioning based on current prices, capacity, and pod needs.

This direct model is why Karpenter can do things CAS structurally cannot: right-size instances per pod, detect configuration drift, actively consolidate underused nodes, and handle Spot interruptions without additional components.

The mental model: Cluster Autoscaler is an ASG manager that happens to respond to Kubernetes scheduling. Karpenter is a Kubernetes-native provisioner that happens to use EC2. Same goal, opposite starting points.

What the Differences Mean in Practice

Provisioning Speed: 60 Seconds vs 5 Minutes

CAS detects pending pods, evaluates ASGs, calls the ASG API to increase desired capacity, the ASG evaluates its launch template, requests an EC2 instance, the instance boots, and the kubelet registers. Each step adds latency. Karpenter collapses this to: detect pending pods, compute optimal instance, call EC2 Fleet API, instance boots, kubelet registers. Fewer hops, fewer API calls, no intermediary evaluation.

This matters most during sudden scale-ups — a burst of queue messages triggering 20 new pods, a traffic spike doubling your API replicas, or a CI pipeline scheduling 50 build jobs. With CAS, those pods sit Pending for minutes. With Karpenter, they're running in about a minute.

Instance Selection: Fixed vs Dynamic

With CAS, you pre-define instance types per ASG. If your ASG is configured for m5.xlarge, every node from that group is an m5.xlarge — even when the pending pod only needs 500m CPU and 512Mi. You're paying for 4 vCPU and 16 GiB to run a pod that needs a fraction of that.

Karpenter evaluates the pod's actual resource requests and picks the cheapest instance that fits. A pod requesting 500m CPU gets a m5.large (2 vCPU). A pod requesting 4 CPU gets an m5.xlarge. A GPU workload gets a g5.xlarge. The decision is per-pod, per-moment, based on current pricing and availability.

Consolidation: Passive Removal vs Active Replacement

CAS scale-down works by identifying nodes that are underutilised (below 50% by default), checking if their pods can be rescheduled elsewhere, and removing the node after a cooldown period. It's passive — it only removes nodes, it never replaces them with better-sized alternatives.

Karpenter's consolidation is active. It doesn't just ask "is this node underused?" — it asks "could these pods fit on fewer or cheaper nodes?" If three m5.xlarge nodes are each running two small pods, Karpenter provisions one m5.large, moves the pods, and terminates all three. The workload keeps running; the infrastructure shrinks. See Karpenter on EKS: Consolidation for detailed examples.

Drift Detection: Manual Pipelines vs Automatic

CAS has no concept of drift. When AWS publishes a new EKS-optimised AMI, your existing nodes keep running the old one. To update, you modify the launch template, trigger a rolling replacement, and wait for nodes to cycle. Most teams build CI/CD pipelines specifically for this — an AMI tracker, a Terraform apply, a rolling update, and monitoring to confirm success.

Karpenter detects AMI drift automatically. Set alias: al2023@latest in your EC2NodeClass, and when a new AMI is published, Karpenter marks affected nodes as drifted and gracefully replaces them — provisioning a new node first, then draining the old one. No pipeline, no manual intervention. See Karpenter on EKS: Drift Detection for the full flow.

Spot Handling: Bolted-On vs Built-In

CAS needs the AWS Node Termination Handler (NTH) — a DaemonSet you install on every node — to detect Spot interruption events and drain pods before termination. NTH is a separate project to install, configure, upgrade, and monitor. When an interruption happens, NTH drains the node, pods go Pending, and only then does CAS start provisioning a replacement. See EKS Spot Instances: Handling Interruptions for the NTH setup.

Karpenter watches for interruption events natively via SQS. When an interruption is detected, it pre-provisions a replacement node before draining the affected one. By the time pods are evicted, the new node is already ready. It also responds to rebalance recommendations — an early signal that arrives minutes before the actual interruption — giving even more time for graceful migration. See Karpenter on EKS: Spot Interruptions for details.

When Cluster Autoscaler Is the Right Choice

Karpenter gets the better headline numbers, but CAS isn't obsolete. There are legitimate scenarios where it's the better fit:

  • Multi-cloud or cloud-agnostic strategy. CAS is a CNCF project that works on AWS, GCP, Azure, and other providers. If you run Kubernetes across multiple clouds, CAS gives you one autoscaler with the same configuration model everywhere. Karpenter is AWS-only.
  • EKS managed node groups with AMI control. If your organisation mandates managed node groups for compliance or operational reasons, CAS integrates natively. Karpenter provisions self-managed instances — you get more control but lose the managed node group lifecycle.
  • Small, stable clusters with predictable workloads. If you have 2–3 node groups, fixed instance types, no Spot, and workloads that don't change much, CAS works fine. The overhead of managing ASGs is low when there are only a few of them.
  • Existing automation that works. If you have Terraform modules, monitoring dashboards, runbooks, and a team trained on CAS — and scaling isn't a pain point — switching to Karpenter is a cost with no immediate benefit. Don't migrate for migration's sake.
  • Restricted environments. GovCloud, air-gapped clusters, or environments with tight IAM restrictions. CAS needs fewer permissions than Karpenter (no SQS, no EventBridge, no EC2 Fleet API). In environments where every IAM action is audited and approved, CAS's simpler permission model is easier to get through security review.
The honest assessment: If CAS is working and scaling isn't a pain point, there's no urgent reason to migrate. CAS is proven, stable, and well-understood. The case for Karpenter is strongest when you're hitting CAS's limitations — slow provisioning, poor bin-packing, Spot management overhead, or manual AMI patching.

When Karpenter Is the Clear Winner

  • You run Spot Instances. Native interruption handling, pre-provisioned replacements, rebalance recommendation support — Karpenter's Spot story is significantly better than CAS + NTH. If Spot is a meaningful part of your cost strategy, this alone justifies the switch.
  • You need fast provisioning. Workloads that scale rapidly — event-driven processing, CI/CD, auto-scaling APIs — benefit from 60-second node provisioning instead of 3–5 minutes.
  • You care about node utilisation and cost. Active consolidation means Karpenter continuously right-sizes your fleet. CAS leaves underused nodes running until the cooldown expires and conditions are met. Teams typically see 20–40% better node utilisation after switching.
  • You want automated AMI patching. Drift detection eliminates the AMI update pipeline entirely. If your team spends hours per month on node patching, Karpenter gives that time back.
  • You're managing too many ASGs. If you have 5+ ASGs with different instance types, architectures, and capacity types, the operational overhead is real. Karpenter replaces them with 1–3 NodePools.
  • You're starting a new cluster. No legacy to migrate, no existing automation to preserve. Karpenter is simpler to set up from scratch than CAS with well-configured ASGs.
  • You pair with KEDA or other pod autoscalers. KEDA + Karpenter creates a fully event-driven pipeline where pod scaling and node scaling are both instant and automatic.

Migrating from CAS to Karpenter

The good news: you can run both simultaneously. CAS manages your existing ASG-backed node groups while Karpenter manages new workloads on directly-provisioned nodes. This lets you migrate gradually without a big-bang cutover.

The Migration Strategy

# Phase 1: Install Karpenter alongside CAS
1. Install Karpenter (Helm) into your cluster
2. Create NodePool + EC2NodeClass for new workloads
3. Both autoscalers run simultaneously
4. CAS manages existing ASG nodes
5. Karpenter manages new NodePool nodes

# Phase 2: Migrate workloads to Karpenter nodes
6. Start with non-critical workloads (dev, staging, batch)
7. Add labels/selectors to steer pods to Karpenter nodes
8. Monitor provisioning speed, utilisation, cost
9. Gradually move production workloads

# Phase 3: Decommission CAS
10. Scale down ASG node groups to zero
11. Remove Cluster Autoscaler deployment
12. Remove Node Termination Handler DaemonSet
13. Clean up ASGs and launch templates

Running Both: How It Works

When both are running, they both watch for Pending pods. To prevent them from competing:

  • Use taints on Karpenter NodePools. Karpenter nodes get a taint that only pods with the matching toleration can be scheduled on. CAS nodes remain untainted. Workloads you want on Karpenter get the toleration; everything else stays on CAS nodes.
  • Use nodeAffinity to prefer Karpenter nodes. A softer approach — pods prefer Karpenter-provisioned nodes but fall back to CAS nodes if Karpenter can't provision (e.g., hitting limits).
  • Karpenter should not manage ASG-backed nodes. Karpenter ignores nodes it didn't provision. CAS ignores nodes without ASG tags. They coexist cleanly as long as you don't mix their concerns.
# Pod that prefers Karpenter nodes but tolerates CAS nodes
spec:
  affinity:
    nodeAffinity:
      preferredDuringSchedulingIgnoredDuringExecution:
        - weight: 100
          preference:
            matchExpressions:
              - key: karpenter.sh/registered
                operator: Exists

What to Watch During Migration

Metric What to Watch Action
Pod scheduling latency Should drop from minutes to ~60s If not, check NodePool requirements are broad enough
Node utilisation Should increase 20–40% If flat, enable WhenEmptyOrUnderutilized consolidation
Pending pods Should be rare and short-lived If persistent, check NodePool limits aren't hit
Spot interruption recovery Should be near-instant Verify SQS queue receives interruption events
Monthly compute cost Should decrease 20–50% Compare before/after with the same workload baseline
Don't rush Phase 3. Run both autoscalers in parallel for at least 2–4 weeks. Let Karpenter handle real traffic patterns — weekday spikes, weekend lulls, batch processing peaks. Only decommission CAS once you're confident Karpenter handles all scenarios correctly.

EKS Auto Mode: Managed Karpenter

If you're choosing Karpenter for a new cluster, consider EKS Auto Mode. Since December 2024, it runs Karpenter as a managed component of the EKS control plane — AWS handles installation, upgrades, and operational health. You still define NodePool and EC2NodeClass the same way. The tradeoff is less visibility and customisation, but zero Karpenter operational overhead.

EKS Auto Mode is not an option if you need to run a specific Karpenter version, customise controller settings, or use Karpenter features before they're available in the managed version. For most teams starting fresh, it's the simplest path.

The Decision Framework

Skip the feature comparison and ask these questions in order:

  1. Do you run on multiple clouds? → CAS. Karpenter is AWS-only. If you need one autoscaler across AWS, GCP, and Azure, CAS is your only option.
  2. Are you starting a new EKS cluster? → Karpenter. No legacy to migrate, simpler to set up, better defaults. Consider EKS Auto Mode for zero operational overhead.
  3. Do you run Spot Instances? → Strong case for Karpenter. Native interruption handling and pre-provisioned replacements are meaningfully better than CAS + NTH.
  4. Is provisioning speed a bottleneck? → Karpenter. If workloads wait 3–5 minutes for nodes and it affects SLAs or developer experience, the 60-second provisioning is a real improvement.
  5. Are you spending time on AMI patching? → Karpenter. Drift detection automates what currently takes a pipeline and manual oversight.
  6. Is CAS working and scaling isn't a pain point? → Stay on CAS. Migrate when you hit a concrete limitation, not because something newer exists.

Key Takeaways

  1. Different architecture, not just different features. CAS manages ASGs (infrastructure-first). Karpenter provisions instances directly (workload-first). This structural difference is why Karpenter can right-size, consolidate, detect drift, and handle Spot natively.
  2. Karpenter wins on speed, cost, and operational simplicity. 60-second provisioning, active consolidation, drift detection, native Spot handling, and two CRDs instead of multiple ASGs + launch templates + NTH.
  3. CAS wins on portability and maturity. Cloud-agnostic (AWS, GCP, Azure), battle-tested since 2016, works with managed node groups, and has a simpler IAM footprint.
  4. Don't migrate for migration's sake. If CAS is working and scaling isn't a pain point, there's no urgent reason to switch. Migrate when you hit concrete limitations.
  5. Spot workloads are the strongest case for Karpenter. Native interruption handling with pre-provisioned replacements is meaningfully better than CAS + NTH.
  6. You can run both during migration. Install Karpenter alongside CAS, use taints or affinity to steer workloads, and migrate gradually over 2–4 weeks.
  7. EKS Auto Mode is the simplest path for new clusters. Managed Karpenter with zero operational overhead — same CRDs, AWS handles the rest.
