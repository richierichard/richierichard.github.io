EKS Spot Instances: A Practical Guide to Cutting Your Kubernetes Compute Costs

Richie Richard Rajkumar · April 2026 · 14 min read

Your EKS cluster is probably running entirely on on-demand EC2 instances. That's fine for reliability — but you're paying full price for every node, including the ones running stateless workers, batch jobs, and dev workloads that could tolerate interruption. EC2 Spot Instances offer the same compute at up to 90% off the on-demand price. The catch: AWS can reclaim them with a two-minute warning when it needs the capacity back.

This isn't a theoretical exercise. Teams running production EKS clusters routinely save 50–70% on compute by moving interruptible workloads to Spot. The key is knowing which workloads to move, how to configure your autoscaler, and what happens when instances get reclaimed. This guide covers two strategies — hybrid (on-demand base + spot for spikes) and full spot — with configurations for both Cluster Autoscaler and Karpenter.

What Are Spot Instances

EC2 Spot Instances are spare compute capacity that AWS sells at a steep discount — typically 60–90% cheaper than on-demand pricing. The tradeoff is simple: when AWS needs that capacity back (because on-demand or Reserved Instance demand in that pool increases), your Spot Instance gets a two-minute interruption notice, and then it's terminated.

A few things that matter for Kubernetes:

  • Same hardware, same performance. A Spot m5.xlarge is identical to an on-demand m5.xlarge — same CPU, memory, network, and storage. You're not getting degraded instances.
  • Price varies by instance type, AZ, and time. A c5.xlarge in us-east-1a might be $0.05/hr right now and $0.08/hr tomorrow. Diversifying across instance types and availability zones is how you reduce interruption risk.
  • Interruption rates vary widely. Popular instance types in constrained AZs get interrupted more often. Less common instance families (like m5a, m6i, c5a) typically have lower interruption rates because fewer people request them.
  • Capacity pools are independent. Each combination of instance type + AZ is a separate capacity pool. Spreading across 10–15 pools gives you much better availability than concentrating on one or two.
Aspect On-Demand Spot
Pricing Full price ($0.192/hr for m5.xlarge) Up to 90% discount (~$0.058/hr)
Availability Guaranteed — always available Subject to capacity — can be reclaimed
Interruption None — runs until you stop it 2-minute warning before termination
Best for Databases, controllers, stateful workloads Stateless APIs, workers, batch, CI/CD, dev/staging

Option 1: Hybrid — On-Demand Base + Spot for Spikes

This is the most common production strategy. You run a stable base of on-demand nodes for workloads that cannot tolerate interruption — databases, stateful services, cluster-critical controllers, monitoring. Then you add Spot nodes for everything else: stateless API replicas, background workers, batch processing, CI/CD runners, and dev/staging environments.

The mental model is simple:

  • On-demand nodes handle your constant baseline load — the pods that must always be running, no matter what.
  • Spot nodes handle spike load and interruptible workloads — the pods that can be rescheduled if a node disappears.

You separate these using taints, tolerations, and node selectors. On-demand nodes run normally. Spot nodes get a taint so that only pods that explicitly opt in get scheduled there.

With Cluster Autoscaler

Cluster Autoscaler works with Auto Scaling Groups (ASGs). You create separate ASGs for on-demand and Spot, and the autoscaler scales each independently based on pending pods.

On-demand node group:

# Auto Scaling Group for on-demand nodes
# Instance type: fixed (e.g., m5.xlarge)
# Min/Max: sized for your baseline workload

Node Group: on-demand-base
  Instance Type:  m5.xlarge
  Capacity Type:  ON_DEMAND
  Min Size:       2
  Max Size:       10
  Labels:
    lifecycle: on-demand
    intent:    base-workloads

Spot node group — use multiple instance types for diversification:

# Auto Scaling Group for spot nodes
# Mixed instance types with capacity-optimized allocation

Node Group: spot-burst
  Instance Types: m5.xlarge, m5a.xlarge, m5d.xlarge, m6i.xlarge,
                   c5.xlarge, c5a.xlarge, c5d.xlarge, c6i.xlarge,
                   r5.xlarge, r5a.xlarge
  Capacity Type:  SPOT
  Min Size:       0
  Max Size:       50
  Allocation Strategy: capacity-optimized
  Labels:
    lifecycle: spot
    intent:    burst-workloads
  Taints:
    - key:    lifecycle
      value:  spot
      effect: NoSchedule

Cluster Autoscaler needs to know how to expand Spot ASGs. Use the --expander=least-waste strategy — it picks the ASG that will have the least idle resources after the pending pods are scheduled:

# Cluster Autoscaler deployment args
args:
  - --cloud-provider=aws
  - --node-group-auto-discovery=asg:tag=k8s.io/cluster-autoscaler/enabled,k8s.io/cluster-autoscaler/my-cluster
  - --expander=least-waste
  - --skip-nodes-with-local-storage=false
  - --balance-similar-node-groups=true   # Spreads across AZs
  - --skip-nodes-with-system-pods=false
Why capacity-optimized? The capacity-optimized allocation strategy tells AWS to launch Spot Instances from the pools with the most available spare capacity — not the cheapest pools. This significantly reduces interruption rates because pools with more capacity are less likely to be reclaimed. The old lowest-price strategy saved a few extra cents but led to much higher interruption rates.

With Karpenter

Karpenter doesn't use ASGs. It evaluates pending pods in real time, computes the cheapest instance type that fits, and provisions individual EC2 instances directly. This is fundamentally different from Cluster Autoscaler — there are no pre-defined node groups to scale. Karpenter decides the instance type, AZ, and purchase option at launch time.

On-demand NodePool for stable workloads:

apiVersion: karpenter.sh/v1
kind: NodePool
metadata:
  name: on-demand-base
spec:
  template:
    metadata:
      labels:
        lifecycle: on-demand
        intent: base-workloads
    spec:
      requirements:
        - key: karpenter.sh/capacity-type
          operator: In
          values: ["on-demand"]
        - key: node.kubernetes.io/instance-type
          operator: In
          values: ["m5.xlarge", "m5.2xlarge", "m6i.xlarge", "m6i.2xlarge"]
        - key: topology.kubernetes.io/zone
          operator: In
          values: ["eu-central-1a", "eu-central-1b", "eu-central-1c"]
      nodeClassRef:
        group: karpenter.k8s.aws
        kind: EC2NodeClass
        name: default
  limits:
    cpu: "100"
    memory: 400Gi

Spot NodePool for burst and interruptible workloads:

apiVersion: karpenter.sh/v1
kind: NodePool
metadata:
  name: spot-burst
spec:
  template:
    metadata:
      labels:
        lifecycle: spot
        intent: burst-workloads
    spec:
      taints:
        - key: lifecycle
          value: spot
          effect: NoSchedule
      requirements:
        - key: karpenter.sh/capacity-type
          operator: In
          values: ["spot"]
        - key: node.kubernetes.io/instance-type
          operator: In
          values:
            - "m5.xlarge"
            - "m5a.xlarge"
            - "m6i.xlarge"
            - "c5.xlarge"
            - "c5a.xlarge"
            - "c6i.xlarge"
            - "r5.xlarge"
            - "r5a.xlarge"
        - key: topology.kubernetes.io/zone
          operator: In
          values: ["eu-central-1a", "eu-central-1b", "eu-central-1c"]
      nodeClassRef:
        group: karpenter.k8s.aws
        kind: EC2NodeClass
        name: default
  disruption:
    consolidationPolicy: WhenEmptyOrUnderutilized
    consolidateAfter: 30s
  limits:
    cpu: "500"
    memory: 2000Gi

Now, your pod specs just need the right toleration and node selector:

# Pod that opts into spot nodes
spec:
  tolerations:
    - key: lifecycle
      value: spot
      operator: Equal
      effect: NoSchedule
  nodeSelector:
    lifecycle: spot
Karpenter's advantage here: Cluster Autoscaler scales ASGs — it picks which group to scale, not which instance type. Karpenter evaluates the pending pod's resource requests and picks the optimal instance type, AZ, and purchase option at launch time. This means tighter bin-packing, less waste, and faster scaling (30–60 seconds vs 3–5 minutes with CAS).

Option 2: Running a Full Spot Cluster

If your entire workload is stateless and can handle interruptions — CI/CD pipelines, dev/staging environments, batch processing, load testing — you can run the entire cluster on Spot. No on-demand nodes at all (except the EKS-managed control plane, which is always on-demand and managed by AWS).

This is where the savings are most dramatic. A cluster that costs $15,000/month on on-demand can drop to $3,000–$5,000/month on Spot.

The rules for full Spot:

  • Diversify aggressively. Use at least 10–15 different instance types across all available AZs. The more pools you spread across, the less likely you are to lose all your capacity at once.
  • Every pod needs a PodDisruptionBudget. No exceptions. If a node gets reclaimed, Kubernetes needs to know how many pods can be down simultaneously.
  • Run enough replicas. If you have a single-replica deployment on Spot, you will have downtime during interruptions. Two replicas minimum, spread across nodes.
  • No local state. Nothing on the node that you can't lose. Logs go to CloudWatch or a logging stack. State goes to RDS, ElastiCache, or S3.

With Cluster Autoscaler

For a full Spot cluster, create multiple ASGs — one per instance family — and let Cluster Autoscaler manage them all:

# Multiple Spot ASGs for maximum diversification
# Cluster Autoscaler discovers them via tags

Node Group: spot-m5-family
  Instance Types: m5.xlarge, m5.2xlarge, m5a.xlarge, m5a.2xlarge
  Capacity Type:  SPOT
  Allocation:     capacity-optimized
  Min: 0  Max: 30

Node Group: spot-c5-family
  Instance Types: c5.xlarge, c5.2xlarge, c5a.xlarge, c5a.2xlarge
  Capacity Type:  SPOT
  Allocation:     capacity-optimized
  Min: 0  Max: 30

Node Group: spot-m6i-family
  Instance Types: m6i.xlarge, m6i.2xlarge, m6g.xlarge, m6g.2xlarge
  Capacity Type:  SPOT
  Allocation:     capacity-optimized
  Min: 0  Max: 30
# Cluster Autoscaler config for full-spot
args:
  - --cloud-provider=aws
  - --node-group-auto-discovery=asg:tag=k8s.io/cluster-autoscaler/enabled,k8s.io/cluster-autoscaler/my-cluster
  - --expander=least-waste
  - --balance-similar-node-groups=true
  - --skip-nodes-with-local-storage=false
  - --skip-nodes-with-system-pods=false
  - --max-graceful-termination-sec=120  # Match the 2-min Spot warning

With Karpenter

Karpenter makes full Spot much simpler — you don't need to manage multiple ASGs. A single NodePool can handle the entire cluster:

apiVersion: karpenter.sh/v1
kind: NodePool
metadata:
  name: full-spot
spec:
  template:
    metadata:
      labels:
        lifecycle: spot
    spec:
      requirements:
        - key: karpenter.sh/capacity-type
          operator: In
          values: ["spot"]
        - key: kubernetes.io/arch
          operator: In
          values: ["amd64"]
        - key: node.kubernetes.io/instance-type
          operator: In
          values:
            - "m5.xlarge"
            - "m5a.xlarge"
            - "m5.2xlarge"
            - "m6i.xlarge"
            - "m6i.2xlarge"
            - "c5.xlarge"
            - "c5a.xlarge"
            - "c5.2xlarge"
            - "c6i.xlarge"
            - "r5.xlarge"
            - "r5a.xlarge"
            - "r5.2xlarge"
        - key: topology.kubernetes.io/zone
          operator: In
          values: ["eu-central-1a", "eu-central-1b", "eu-central-1c"]
      nodeClassRef:
        group: karpenter.k8s.aws
        kind: EC2NodeClass
        name: default
  disruption:
    consolidationPolicy: WhenEmptyOrUnderutilized
    consolidateAfter: 30s
  limits:
    cpu: "1000"
    memory: 4000Gi

Karpenter automatically applies the capacity-optimized allocation strategy for Spot — it evaluates available capacity across all the instance types and AZs you've listed and picks the combination with the deepest pool. No ASG configuration needed.

Handling Spot Interruptions

When AWS needs capacity back, your Spot Instance gets a two-minute warning. What happens next depends on your autoscaler.

Cluster Autoscaler: AWS Node Termination Handler

Cluster Autoscaler has no built-in awareness of Spot interruptions. You need the AWS Node Termination Handler (NTH) — a DaemonSet that runs on every node, watches for EC2 interruption events, and gracefully drains the node before it's terminated.

NTH can operate in two modes:

  • Instance Metadata Service (IMDS) mode — the DaemonSet polls the instance metadata endpoint for interruption notices. Simpler to set up, but only detects Spot interruptions and scheduled events.
  • Queue mode — uses an SQS queue that receives EventBridge events for Spot interruptions, rebalance recommendations, scheduled maintenance, and instance state changes. More comprehensive and the recommended approach for production.

The interruption flow with NTH:

# What happens during a Spot interruption with NTH:

1. AWS sends 2-minute interruption notice
2. NTH detects the notice (via IMDS poll or SQS event)
3. NTH cordons the node      → no new pods scheduled
4. NTH drains the node       → existing pods get SIGTERM
5. Pods respect their terminationGracePeriodSeconds
6. Pods become Pending       → Cluster Autoscaler provisions a new node
7. Scheduler places pods on the new node
Set terminationGracePeriodSeconds wisely. Your pods need to shut down within the two-minute window. If your pod takes 90 seconds to drain connections, that leaves only 30 seconds for the node drain to begin and the SIGTERM to propagate. Keep it under 90 seconds as a rule of thumb.

Karpenter: Native Interruption Handling

Karpenter handles Spot interruptions natively — no separate DaemonSet needed. It watches for EC2 interruption warnings, rebalance recommendations, and instance health events through the EC2 API and EventBridge. When an interruption is detected, Karpenter:

  1. Cordons the affected node
  2. Pre-provisions a replacement node before draining — so there's capacity ready
  3. Drains the old node once the replacement is ready
  4. Deletes the old node object from the cluster

This is a significant improvement over the NTH + CAS flow. With Karpenter, the replacement node is already launching while the old node is being drained. With CAS + NTH, the replacement only starts provisioning after the pods become Pending — adding 3–5 minutes of delay.

# Karpenter interruption handling — no configuration needed
# These capabilities are built in when you deploy Karpenter:

✓ Spot Instance interruption warnings    (2-min notice)
✓ EC2 rebalance recommendations           (early warning)
✓ Scheduled change health events          (maintenance)
✓ Instance state change notifications     (stopped/terminated)

# Karpenter requires these IAM permissions for interruption handling:
ec2:DescribeInstanceStatus
sqs:ReceiveMessage         # If using SQS-based events
sqs:DeleteMessage
events:PutRule             # EventBridge rules
events:PutTargets

Best Practices for Production

1. Pod Disruption Budgets — Non-Negotiable

A PodDisruptionBudget (PDB) tells Kubernetes how many pods in a deployment can be down simultaneously. Without PDBs, a Spot interruption could take down all replicas of a service at once if they're on the same node.

apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: api-pdb
spec:
  minAvailable: "50%"   # At least half must stay running
  selector:
    matchLabels:
      app: api

2. Topology Spread Constraints — Don't Put All Eggs on One Node

Spread your pod replicas across nodes and AZs. If one Spot node gets reclaimed, only a fraction of your replicas are affected:

spec:
  topologySpreadConstraints:
    - maxSkew: 1
      topologyKey: kubernetes.io/hostname
      whenUnsatisfiable: DoNotSchedule
      labelSelector:
        matchLabels:
          app: api
    - maxSkew: 1
      topologyKey: topology.kubernetes.io/zone
      whenUnsatisfiable: ScheduleAnyway
      labelSelector:
        matchLabels:
          app: api

3. Instance Diversification — The Single Most Important Thing

This cannot be overstated: the more instance types and AZs you use, the lower your interruption rate. A single instance type in a single AZ is a single point of failure. Spread across 10+ instance types and all available AZs.

Good instance families to mix for general-purpose Kubernetes workloads:

Family Type Typical Spot Discount Notes
m5 / m5a / m5d General purpose 60–75% Balanced CPU/memory, good baseline
m6i / m6a General purpose (newer) 65–80% Better price-performance than m5
c5 / c5a / c5d Compute optimized 60–70% Good for CPU-bound workloads
c6i / c6a Compute optimized (newer) 65–80% Lower interruption rates (newer generation)
r5 / r5a Memory optimized 60–75% For memory-heavy workloads

4. Capacity-Optimized Allocation — Always

Whether you're using Cluster Autoscaler with ASGs or Karpenter, always use the capacity-optimized allocation strategy. It launches instances from pools with the most spare capacity, which directly reduces interruption rates. Karpenter does this by default. For CAS, set it on your ASG's Spot allocation strategy.

5. Set Resource Requests Accurately

Both autoscalers make scaling decisions based on pod resource requests. If your requests are inflated, you'll overprovision (wasting money). If they're too low, nodes will be overcommitted and pods will get OOMKilled or throttled. Use tools like the Vertical Pod Autoscaler (VPA) in recommendation mode to right-size your requests.

6. Use Spot-Friendly Architectures

  • Stateless services with multiple replicas — if one pod dies, the others absorb the traffic
  • Queue-based workers — if a worker dies mid-job, the message goes back to the queue and another worker picks it up
  • Checkpointing for long-running jobs — ML training saves checkpoints to S3; if interrupted, it resumes from the last checkpoint
  • Graceful shutdown handlers — trap SIGTERM, finish in-flight requests, close connections cleanly

Cluster Autoscaler vs Karpenter: Which Should You Use

Aspect Cluster Autoscaler Karpenter
Scaling model Scales ASGs up/down Provisions individual instances directly
Instance type selection Fixed per ASG — you pre-define the types Dynamic — picks optimal type per pod at launch
Time to new node 3–5 minutes (ASG → EC2 → register → ready) 30–60 seconds (direct EC2 fleet API call)
Spot interruption handling Requires AWS Node Termination Handler (separate DaemonSet) Native — built-in, pre-provisions replacement
Bin packing Picks which ASG to scale, not which instance fits best Right-sizes instance to pending pod needs
Consolidation Scale-down only (removes underused nodes) Active consolidation — replaces nodes with cheaper/smaller ones
Operational complexity Manage multiple ASGs, launch templates, NTH NodePool + EC2NodeClass — less infrastructure to manage
Maturity Battle-tested, widely deployed since 2018 GA since 2023, rapidly adopted, AWS-maintained
If you're starting fresh, use Karpenter. It's simpler to configure, faster to scale, handles interruptions natively, and produces tighter bin-packing. If you're already running Cluster Autoscaler and it's working, there's no rush to migrate — but Spot is where Karpenter's advantages are most pronounced.

A Note on EKS Auto Mode

Since December 2024, EKS Auto Mode runs Karpenter as a managed, off-cluster component. You don't install, configure, or upgrade Karpenter yourself — AWS manages it as part of the EKS control plane. Spot Instance support, interruption handling, and consolidation all work out of the box.

If you're setting up a new cluster and want Spot with minimal operational overhead, EKS Auto Mode is worth evaluating. You define your NodePool and EC2NodeClass resources the same way, but AWS handles the Karpenter lifecycle. The tradeoff is less customisation and visibility into the autoscaler internals — but for most teams, the defaults are good enough.

We'll cover EKS Auto Mode in depth in a dedicated article.

What the Numbers Look Like

Here's a realistic before-and-after for a mid-size production EKS cluster:

Metric Before (All On-Demand) After (Hybrid Spot)
Monthly compute spend $18,000 $7,200
On-demand nodes 25 × m5.xlarge 8 × m5.xlarge (baseline)
Spot nodes 0 17 × mixed (m5, c5, m6i, r5)
Avg interruptions/week 0 2–4 (handled gracefully)
Service availability 99.95% 99.93% (negligible difference)
Annual savings — ~$130,000

The 0.02% availability difference comes from the brief window during interruption handling — pods are in-flight between old and new nodes. With proper PDBs, topology spread, and multiple replicas, users don't notice it.

Key Takeaways

  1. Spot Instances offer 60–90% savings on EC2 compute — same hardware, same performance, with the caveat that AWS can reclaim capacity with a 2-minute notice
  2. Hybrid is the safe starting point: on-demand nodes for databases, controllers, and stateful workloads; Spot nodes for stateless APIs, workers, batch jobs, and dev/staging
  3. Full Spot works for fully stateless environments — CI/CD, dev/staging, batch processing, load testing — where every pod can tolerate interruption
  4. Instance diversification is the single most important practice — spread across 10+ instance types and all available AZs to minimize interruption risk
  5. Always use capacity-optimized allocation — it selects Spot pools with the most spare capacity, reducing interruptions
  6. Karpenter handles Spot better than Cluster Autoscaler — native interruption handling, faster scaling, tighter bin-packing, and active consolidation. CAS works but needs the Node Termination Handler as a separate component
  7. PodDisruptionBudgets and topology spread constraints are mandatory — they ensure graceful handling when nodes disappear
  8. EKS Auto Mode manages Karpenter for you — if you want Spot with minimal ops overhead, it's the simplest path
