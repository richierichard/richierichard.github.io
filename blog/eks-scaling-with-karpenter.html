Karpenter on EKS: Right-Sized Nodes in 60 Seconds

Richie Richard Rajkumar · May 2026 · 15 min read

Your EKS cluster probably scales nodes through Auto Scaling Groups. You've defined launch templates, configured minimum and maximum sizes, picked instance types upfront, and hoped the autoscaler picks the right group when pods go Pending. When you need a new AMI, you update the launch template, trigger a rolling update, and wait. When nodes sit underused, you wait for the scale-down cooldown. Every change is a change to AWS infrastructure, not Kubernetes resources.

Karpenter replaces all of that with two Kubernetes custom resources: NodePool and EC2NodeClass. You declare what your workloads need — instance families, capacity types, availability zones — and Karpenter provisions the exact right node in 60 seconds. When the node drifts from your desired spec, it gets replaced automatically. When nodes are underused, they get consolidated without waiting. This guide walks through setting up Karpenter on EKS and the four features that make it genuinely different: drift detection, consolidation, expiration, and native Spot interruption handling.

Why Node Scaling Needed a Rethink

Traditional node autoscaling in Kubernetes is infrastructure-first. You define node groups ahead of time — each with a fixed set of instance types, a minimum and maximum size, and a launch template. When pods can't schedule, the autoscaler picks one of these pre-defined groups and adds a node from it. When nodes are idle, it waits for a cooldown period and then removes them.

This model has real friction points that anyone who's operated EKS at scale has felt:

  • Instance types are chosen at group creation time, not at scheduling time. You pick m5.xlarge for your node group, and every node that comes out of it is an m5.xlarge — even when the pending pod only needs 500m CPU and 512Mi memory. You're paying for a 4 vCPU / 16 GiB node to run a pod that needs a fraction of that.
  • Scaling is slow. The controller detects pending pods, decides which group to scale, calls the ASG API, waits for the EC2 instance to launch, waits for it to join the cluster, and then the pod gets scheduled. End-to-end: 3–5 minutes on a good day.
  • AMI updates require rolling infrastructure changes. When a new EKS-optimised AMI is released — security patches, kernel updates, runtime upgrades — you update the launch template, then trigger a rolling replace. Nodes running the old AMI stay in the cluster until you explicitly replace them.
  • Consolidation is passive. Underused nodes only get removed if they fall below a utilisation threshold and the cooldown timer has expired. The controller won't actively move workloads onto fewer, better-fitted nodes. It just removes empty or nearly-empty nodes and hopes the scheduler does the rest.
  • Spot interruptions need a separate component. You need to install and maintain a DaemonSet (AWS Node Termination Handler) on every node to detect Spot interruption warnings and drain pods gracefully. It's another moving part that can break.

Karpenter was designed at AWS to eliminate these friction points. Instead of managing node groups, you declare intent — "I need nodes from these instance families, in these AZs, using on-demand or Spot" — and Karpenter handles every decision from there: which instance type, which AZ, when to launch, when to replace, when to consolidate.

The Two Resources That Replace Everything

Karpenter's entire configuration lives in two Kubernetes custom resources. No ASGs, no launch templates, no managed node groups.

NodePool — What Karpenter Is Allowed to Provision

A NodePool defines the constraints and policies for a group of nodes. Think of it as the answer to: "what kinds of nodes can Karpenter create, and how should it manage them?"

  • Requirements — instance types, capacity types (on-demand/spot), architectures, availability zones
  • Limits — maximum total CPU and memory Karpenter can provision for this pool
  • Disruption policy — how and when Karpenter consolidates or replaces nodes
  • Expiration — how long a node can live before it gets recycled

EC2NodeClass — The AWS-Specific Configuration

An EC2NodeClass defines everything AWS-specific that every node in the pool gets. This is the configuration that would normally live in a launch template:

  • AMI selection — which AMI to use (EKS-optimised, Bottlerocket, custom)
  • Subnets — which subnets nodes can be placed in
  • Security groups — which security groups to attach
  • Instance profile — the IAM instance profile for the node
  • Block device mappings — disk size and type
  • Tags — AWS resource tags for cost allocation and management
NodePool is Kubernetes-native. EC2NodeClass is AWS-native. This separation is intentional. The NodePool defines scheduling constraints using Kubernetes concepts (requirements, taints, labels). The EC2NodeClass defines infrastructure using AWS concepts (AMIs, subnets, security groups). If you ever migrate to a different cloud, only the NodeClass changes — the NodePool stays the same.

Installing Karpenter on EKS

Karpenter runs as a Deployment in your cluster — typically two replicas for high availability. It needs an IAM role with permissions to manage EC2 instances, and an SQS queue for interruption events.

Prerequisites

Before installing, you need:

  • An EKS cluster (v1.25+) with at least one managed node group for Karpenter itself to run on
  • An IAM role for Karpenter with EC2, SQS, SSM, EKS, and IAM permissions
  • An SQS queue for receiving EC2 interruption and rebalance events
  • An instance profile for the nodes Karpenter will provision

The Karpenter documentation provides a CloudFormation template that creates all the IAM resources. For production, use Terraform or your IaC tool of choice.

Helm Install

# Add the Karpenter Helm repo
$ helm registry logout public.ecr.aws
$ helm upgrade --install karpenter oci://public.ecr.aws/karpenter/karpenter \
    --version "1.1.1" \
    --namespace "kube-system" \
    --set "settings.clusterName=my-cluster" \
    --set "settings.interruptionQueue=my-cluster-karpenter" \
    --set controller.resources.requests.cpu=1 \
    --set controller.resources.requests.memory=1Gi \
    --set controller.resources.limits.memory=1Gi \
    --set "serviceAccount.annotations.eks\.amazonaws\.com/role-arn=arn:aws:iam::123456789012:role/KarpenterControllerRole" \
    --wait

Verify the installation:

$ kubectl get pods -n kube-system -l app.kubernetes.io/name=karpenter
NAME                         READY   STATUS    AGE
karpenter-5d7f8c4b6-j2k9p   1/1     Running   30s
karpenter-5d7f8c4b6-m8n3q   1/1     Running   30s

Two replicas, leader-elected. Karpenter is now watching for NodePool and EC2NodeClass resources.

Configuring Your NodePool

Here's a production-ready NodePool that covers most general-purpose workloads:

apiVersion: karpenter.sh/v1
kind: NodePool
metadata:
  name: general-purpose
spec:
  template:
    metadata:
      labels:
        team: platform
        managed-by: karpenter
    spec:
      requirements:
        - key: kubernetes.io/arch
          operator: In
          values: ["amd64"]
        - key: karpenter.sh/capacity-type
          operator: In
          values: ["on-demand", "spot"]
        - key: node.kubernetes.io/instance-type
          operator: In
          values:
            - "m5.large"
            - "m5.xlarge"
            - "m5.2xlarge"
            - "m5a.large"
            - "m5a.xlarge"
            - "m6i.large"
            - "m6i.xlarge"
            - "m6i.2xlarge"
            - "c5.large"
            - "c5.xlarge"
            - "c6i.large"
            - "c6i.xlarge"
        - key: topology.kubernetes.io/zone
          operator: In
          values: ["eu-central-1a", "eu-central-1b", "eu-central-1c"]
      nodeClassRef:
        group: karpenter.k8s.aws
        kind: EC2NodeClass
        name: default
      expireAfter: 720h             # Recycle nodes every 30 days
  disruption:
    consolidationPolicy: WhenEmptyOrUnderutilized
    consolidateAfter: 60s
  limits:
    cpu: "200"                    # Max 200 vCPUs across all nodes in this pool
    memory: 800Gi                # Max 800 GiB memory
  weight: 50                      # Priority when multiple NodePools match

A few things to note:

  • requirements define what Karpenter is allowed to provision — not what it will provision. Karpenter picks the optimal combination at launch time based on what the pending pods actually need.
  • Mixing capacity types — listing both on-demand and spot lets Karpenter choose. It prefers Spot for cost optimisation but falls back to on-demand when Spot capacity is unavailable. If you need to guarantee on-demand for specific workloads, create a separate NodePool with only on-demand and use taints/tolerations.
  • limits are your safety net. They prevent Karpenter from provisioning a runaway number of nodes if something goes wrong (a deployment with replicas: 10000 due to a misconfiguration, for example).
  • weight determines priority when multiple NodePools can satisfy a pod. Higher weight means Karpenter tries this pool first. Useful for preferring Spot over on-demand.

Configuring the EC2NodeClass

apiVersion: karpenter.k8s.aws/v1
kind: EC2NodeClass
metadata:
  name: default
spec:
  role: KarpenterNodeRole        # IAM role for provisioned nodes
  amiSelectorTerms:
    - alias: al2023@latest        # Latest Amazon Linux 2023 EKS-optimised AMI
  subnetSelectorTerms:
    - tags:
        karpenter.sh/discovery: my-cluster
  securityGroupSelectorTerms:
    - tags:
        karpenter.sh/discovery: my-cluster
  blockDeviceMappings:
    - deviceName: /dev/xvda
      ebs:
        volumeSize: 100Gi
        volumeType: gp3
        encrypted: true
        deleteOnTermination: true
  tags:
    Environment: production
    ManagedBy: karpenter

The amiSelectorTerms with alias: al2023@latest tells Karpenter to always use the latest EKS-optimised Amazon Linux 2023 AMI. When AWS publishes a new AMI — security patches, kernel updates, runtime upgrades — Karpenter detects that existing nodes are running an older AMI and marks them as drifted. More on this in the drift detection section.

Subnets and security groups are selected by tag, not by ID. This means you can add or remove subnets and security groups by updating tags — Karpenter picks up the changes automatically without any resource modifications.

How Provisioning Works

When a pod can't schedule because there's no node with enough capacity, here's what Karpenter does — and how fast it does it:

# Karpenter provisioning flow — end to end ~60 seconds

1. Pod goes Pending              # Scheduler can't find a node
2. Karpenter detects the pod     # Watches the scheduler queue
3. Evaluates pod requirements    # CPU, memory, GPU, tolerations, affinity
4. Batches with other pending    # Groups pods that can share a node
5. Selects optimal instance      # Cheapest type that fits all batched pods
6. Calls EC2 Fleet API           # Direct API call — no ASG intermediary
7. Instance launches              # ~30 seconds for the EC2 instance
8. Kubelet registers the node    # Node becomes Ready
9. Pods scheduled and running    # Total: ~60 seconds

Step 4 is where the magic happens. Karpenter doesn't launch one node per pending pod. It batches pods that arrive around the same time and computes the minimum number of nodes that can serve all of them. If you have three pods pending — one needing 2 CPU, one needing 1 CPU, and one needing 500m CPU — Karpenter might launch a single m5.xlarge (4 vCPU) instead of three separate instances.

This is fundamentally different from ASG-based scaling, where the autoscaler asks "which pre-defined group should I add a node to?" Karpenter asks "what is the cheapest EC2 instance that satisfies these specific pods right now?" The answer changes with every provisioning decision, because Spot pricing, capacity, and pod requirements are all dynamic.

Why 60 seconds instead of 3–5 minutes? Karpenter calls the EC2 Fleet API directly — there's no ASG resize, no launch template evaluation, no Auto Scaling Group cooldown to wait through. The EC2 instance launch itself takes ~30 seconds. The remaining time is kubelet registration and the pod being scheduled. It's the same EC2 underneath — the difference is skipping the ASG intermediary.

Drift Detection

This is the feature that fundamentally changes how you think about node lifecycle management. Drift detection means Karpenter continuously compares your running nodes against the current desired state defined in your NodePool and EC2NodeClass. When they diverge, Karpenter marks the node as drifted and gracefully replaces it.

What Triggers Drift

Drift Type What Changed What Karpenter Does
AMI drift New EKS-optimised AMI published by AWS Provisions new node with latest AMI, drains old node
Security group drift You changed which security groups are tagged for Karpenter Replaces node with updated security groups
Subnet drift You added or removed subnets from the selector New nodes use updated subnet list
NodePool requirement drift You changed instance types, capacity types, or AZs in the NodePool Nodes not matching new requirements get replaced
Block device drift You changed disk size or type in EC2NodeClass New nodes get the updated disk configuration

AMI Drift in Practice

This is the most common drift scenario. You set alias: al2023@latest in your EC2NodeClass. AWS publishes a new EKS-optimised AMI with a critical security patch. Here's what happens — without you doing anything:

# AMI drift flow — fully automatic

1. AWS publishes new EKS AMI     # e.g., kernel CVE patch
2. Karpenter detects the new AMI  # Polls SSM Parameter Store
3. Compares against running nodes # Node ami-0a1b2c ≠ latest ami-0x9y8z
4. Marks nodes as Drifted        # Status condition on the NodeClaim
5. Provisions replacement node    # With the new AMI, same spec
6. Waits for replacement Ready    # New node joins the cluster
7. Drains the old node            # Respects PodDisruptionBudgets
8. Terminates old instance        # Node with outdated AMI is gone

# Result: your fleet is patched without a single manual step

No Terraform apply. No ASG rolling update. No pipeline to trigger. Karpenter handles the entire node replacement lifecycle, and it respects your PodDisruptionBudgets so pods are never disrupted beyond what you've allowed.

This replaces entire patching pipelines. Most teams have a CI/CD pipeline that detects new AMIs, updates launch templates, and triggers rolling replacements across node groups. With Karpenter, you declare al2023@latest once, and your nodes stay current automatically. The pipeline, the AMI tracking job, and the rolling update logic all become unnecessary.

Consolidation

Consolidation is how Karpenter actively optimises your cluster's node usage. It doesn't just remove empty nodes — it analyses whether workloads can be reorganised onto fewer or cheaper nodes, and then makes it happen.

Two Consolidation Policies

Policy Behaviour Best For
WhenEmpty Only removes nodes that have zero non-daemonset pods Conservative — minimises disruption
WhenEmptyOrUnderutilized Removes empty nodes AND replaces underused ones with smaller/cheaper instances Aggressive — maximises cost savings

The WhenEmptyOrUnderutilized policy is where the real value is. Here's a scenario:

# Before consolidation:
Node A (m5.xlarge — 4 vCPU, 16 GiB):  using 1 vCPU, 2 GiB  (25% utilised)
Node B (m5.xlarge — 4 vCPU, 16 GiB):  using 800m, 1.5 GiB   (20% utilised)
Node C (m5.xlarge — 4 vCPU, 16 GiB):  using 500m, 1 GiB     (12% utilised)

# Total: 3 × m5.xlarge = $0.576/hr, ~57% of capacity wasted

# After consolidation:
Node X (m5.large — 2 vCPU, 8 GiB):   using 2.3 vCPU, 4.5 GiB  (~75% utilised)

# Total: 1 × m5.large = $0.096/hr — 83% cost reduction

Karpenter computed that all pods from Nodes A, B, and C fit on a single m5.large. It provisions the new node, drains the old ones (respecting PDBs), and terminates them. Three underused m5.xlarge instances become one right-sized m5.large.

The consolidateAfter parameter controls how quickly this happens:

disruption:
  consolidationPolicy: WhenEmptyOrUnderutilized
  consolidateAfter: 60s    # Start consolidating 60s after a node becomes underused

Set this to 60s for aggressive cost optimisation (batch workloads, dev/staging). Set it to 300s or higher for production workloads where you want to avoid frequent pod movements during short traffic dips.

Consolidation is not just removal — it's replacement. Traditional scale-down only deletes empty or underused nodes and hopes the scheduler reschedules pods well. Karpenter actively provisions a better-fitted replacement before draining the old nodes. The workload is always running — it just moves to cheaper infrastructure.

Expiration

The expireAfter field on a NodePool forces node recycling after a fixed duration. Every node provisioned by this pool gets a TTL. When it expires, Karpenter replaces the node — regardless of whether it's drifted, underused, or perfectly healthy.

spec:
  template:
    spec:
      expireAfter: 720h    # 30 days — every node gets replaced monthly

Why Force Replacement of Healthy Nodes?

  • Security patching as a guarantee. Even if AMI drift detection covers most cases, expiration ensures no node lives longer than your maximum tolerance. If drift detection misses something or you're using a custom AMI without automatic updates, expiration is your backstop.
  • Preventing node-level state accumulation. Long-lived nodes accumulate state: stale iptables rules, leaked file descriptors, zombie processes, container runtime artifacts. Periodic recycling gives you a clean slate.
  • Keeping your fleet fresh. Nodes provisioned 90 days ago might be running on older hardware, in less optimal AZs, or with configurations that no longer match your current NodePool. Expiration forces Karpenter to re-evaluate and provision with the latest spec.
  • Compliance. Some security frameworks (SOC 2, PCI-DSS) require periodic infrastructure rotation. Expiration gives you automated compliance without manual patching windows.

The replacement flow is the same as drift: Karpenter provisions a new node, waits for it to become Ready, drains the expiring node (respecting PDBs), and terminates it. Workloads keep running.

# Common expiration values:
expireAfter: 336h     # 14 days — aggressive, good for Spot-heavy pools
expireAfter: 720h     # 30 days — balanced, good default for production
expireAfter: 2160h    # 90 days — conservative, for stable long-running workloads
Expiration + drift detection = belt and suspenders. Drift handles reactive replacement (something changed, replace the node). Expiration handles proactive replacement (nothing changed, but it's time for a fresh node anyway). Together, they ensure your fleet never goes stale.

Handling Spot Interruptions

Karpenter handles Spot interruptions natively. No separate DaemonSet. No additional controller. When you install Karpenter with an SQS queue configured, it automatically watches for EC2 interruption events and acts on them.

What Karpenter Detects

  • Spot Instance interruption warnings — the 2-minute notice before AWS reclaims the instance
  • EC2 rebalance recommendations — an early signal that a Spot Instance is at elevated risk of interruption, often arriving minutes before the actual notice
  • Scheduled change health events — AWS maintenance events that require instance restart or retirement
  • Instance state change notifications — when an instance is stopped or terminated externally

The Interruption Flow

# Spot interruption handling — Karpenter's native flow

1. AWS sends interruption event to SQS    # 2-minute warning
2. Karpenter receives the message           # Polling the SQS queue
3. Cordons the affected node               # No new pods scheduled
4. Pre-provisions a replacement node       # THIS IS THE KEY STEP
5. Waits for replacement to become Ready   # New node joins cluster
6. Drains the old node                     # Pods migrate to the new node
7. Old instance is terminated by AWS        # Spot reclamation completes

# The gap where pods have no node is minimised
# because the replacement is already Ready before draining starts

Step 4 is the critical difference. Karpenter starts launching the replacement node immediately when the interruption event arrives — it doesn't wait for pods to become Pending first. By the time the drain starts, the replacement node is already joining the cluster. This means the window where pods are unscheduled is seconds, not minutes.

Rebalance Recommendations

EC2 rebalance recommendations are an early warning system. AWS sends this signal when a Spot Instance is at elevated risk of interruption — sometimes 5–10 minutes before the actual 2-minute warning. Karpenter acts on these too:

# Rebalance recommendation flow:

1. AWS sends rebalance recommendation     # "This instance may be interrupted soon"
2. Karpenter provisions replacement        # Proactive — before the 2-min warning
3. Replacement becomes Ready               # Plenty of time — no urgency yet
4. Karpenter drains the at-risk node       # Graceful migration
5. If actual interruption comes → handled  # Pods already moved
6. If no interruption → node was replaced  # Still a good outcome

This proactive approach means most Spot interruptions are handled before the 2-minute warning even arrives. Your pods migrate gracefully to a new node while there's still plenty of time, rather than racing against a 2-minute clock.

No Node Termination Handler needed. With ASG-based scaling, you install the AWS Node Termination Handler as a DaemonSet on every node, configure it to watch for interruption events, and maintain it as a separate component. With Karpenter, all of this is built in. One less DaemonSet, one less thing to upgrade, one less thing to break.

KEDA + Karpenter: The Full Event-Driven Pipeline

Karpenter scales nodes. But what scales the pods? If you're running queue-based workloads — SQS consumers, Kafka processors, batch pipelines — pairing Karpenter with KEDA (Kubernetes Event-Driven Autoscaling) creates a fully event-driven scaling pipeline from queue to compute.

Here's how they work together:

# The full event-driven pipeline: KEDA + Karpenter

1. Messages arrive in SQS                 # External demand signal
2. KEDA reads queue depth                  # 50 messages ÷ 10 threshold = 5 pods
3. KEDA scales Deployment from 0 → 5       # New pods go Pending
4. Karpenter detects Pending pods           # Evaluates resource requirements
5. Karpenter provisions right-sized Spot    # Node ready in ~60 seconds
6. Pods schedule and start processing       # Queue drains
7. Queue empties → KEDA scales to 0         # Pods terminated
8. Karpenter consolidates empty nodes       # Node terminated

# Result: zero compute when idle, instant scale-up on demand

The key insight is that KEDA and Karpenter operate at different layers and complement each other perfectly. KEDA decides how many pods based on the event source (queue depth, Kafka lag, Prometheus metric). Karpenter decides which nodes based on the pods' resource requirements. Neither needs to know about the other — they communicate through the Kubernetes scheduler.

Example: SQS Worker with KEDA and Karpenter

Here's a complete setup. The ScaledObject tells KEDA to scale the worker Deployment based on SQS queue depth. The NodePool tells Karpenter to provision Spot instances for those workers.

# 1. Worker Deployment — starts at 0 replicas (KEDA controls this)
apiVersion: apps/v1
kind: Deployment
metadata:
  name: sqs-worker
spec:
  replicas: 0
  selector:
    matchLabels:
      app: sqs-worker
  template:
    metadata:
      labels:
        app: sqs-worker
        workload-type: batch
    spec:
      tolerations:
        - key: workload-type
          value: batch
          effect: NoSchedule
      nodeSelector:
        workload-type: batch
      containers:
        - name: worker
          image: my-registry/sqs-worker:latest
          resources:
            requests:
              cpu: "500m"
              memory: "512Mi"
# 2. KEDA ScaledObject — scales based on SQS queue depth
apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: sqs-worker-scaler
spec:
  scaleTargetRef:
    name: sqs-worker
  minReplicaCount: 0
  maxReplicaCount: 30
  pollingInterval: 15
  cooldownPeriod: 300
  triggers:
    - type: aws-sqs-queue
      metadata:
        queueURL: "https://sqs.eu-central-1.amazonaws.com/123456789012/order-queue"
        queueLength: "10"
        awsRegion: "eu-central-1"
# 3. Karpenter NodePool — Spot-only pool for batch workers
apiVersion: karpenter.sh/v1
kind: NodePool
metadata:
  name: batch-workers
spec:
  template:
    metadata:
      labels:
        workload-type: batch
    spec:
      taints:
        - key: workload-type
          value: batch
          effect: NoSchedule
      requirements:
        - key: karpenter.sh/capacity-type
          operator: In
          values: ["spot"]
        - key: node.kubernetes.io/instance-type
          operator: In
          values: ["m5.large", "m5a.large", "c5.large", "c5a.large", "m6i.large"]
      nodeClassRef:
        group: karpenter.k8s.aws
        kind: EC2NodeClass
        name: default
      expireAfter: 336h
  disruption:
    consolidationPolicy: WhenEmptyOrUnderutilized
    consolidateAfter: 30s
  limits:
    cpu: "100"
    memory: 400Gi

The taint on the NodePool ensures only batch workers land on these Spot nodes. When the queue is empty, KEDA scales the Deployment to zero, the nodes become empty, and Karpenter consolidates them within 30 seconds. You pay for zero worker compute overnight — not just zero pods, but zero nodes.

This is where the real savings happen. KEDA alone eliminates idle pods. Karpenter alone eliminates overprovisioned nodes. Together, they eliminate idle pods and the nodes they ran on. If your queue-based workloads are idle for 12 hours overnight, that's 12 hours of zero compute cost — not just reduced, but zero.

Production Hardening

1. Disruption Budgets on NodePools

Control how many nodes Karpenter can disrupt simultaneously. This prevents a scenario where Karpenter decides to consolidate or replace half your fleet at once:

spec:
  disruption:
    consolidationPolicy: WhenEmptyOrUnderutilized
    consolidateAfter: 60s
    budgets:
      - nodes: "20%"            # Max 20% of nodes disrupted at once
      - nodes: "0"              # No disruptions during business hours
        schedule: "0 9 * * 1-5" # Monday–Friday 9am
        duration: 8h            # Until 5pm

The schedule-based budget is powerful — you can tell Karpenter "consolidate and replace nodes overnight, but don't touch anything during business hours." This gives you aggressive cost optimisation without risking production stability during peak traffic.

2. The do-not-disrupt Annotation

For pods that absolutely cannot be moved — long-running ML training jobs, database leaders during failover, or batch jobs that don't checkpoint — annotate them:

metadata:
  annotations:
    karpenter.sh/do-not-disrupt: "true"

Karpenter will not drain a node that has a pod with this annotation. The node stays until the pod completes or the annotation is removed. Use this sparingly — overuse defeats the purpose of consolidation.

3. Separate NodePools for Different Workload Classes

Don't put everything in one NodePool. Separate by workload characteristics:

# Critical workloads — on-demand only, no aggressive consolidation
NodePool: critical
  capacity-type: on-demand
  consolidationPolicy: WhenEmpty
  weight: 100

# General workloads — spot preferred, active consolidation
NodePool: general-purpose
  capacity-type: on-demand, spot
  consolidationPolicy: WhenEmptyOrUnderutilized
  weight: 50

# Batch/dev — spot only, aggressive consolidation, short TTL
NodePool: batch
  capacity-type: spot
  consolidationPolicy: WhenEmptyOrUnderutilized
  consolidateAfter: 30s
  expireAfter: 336h
  weight: 10

Use taints and tolerations to steer workloads to the right pool. Critical databases go to the on-demand pool. Stateless APIs go to general-purpose. Batch jobs go to the Spot-only pool.

4. Monitor Karpenter Metrics

Karpenter exposes Prometheus metrics that you should be watching:

# Key Karpenter metrics
karpenter_nodes_total                    # Total nodes managed by Karpenter
karpenter_nodeclaims_disrupted_total     # Nodes disrupted (by reason: drift, consolidation, etc.)
karpenter_pods_startup_duration_seconds  # Time from pod creation to pod running
karpenter_nodeclaims_created_total       # New nodes provisioned
karpenter_nodeclaims_terminated_total    # Nodes removed
karpenter_provisioner_scheduling_duration_seconds # How long scheduling decisions take

The most important metric is karpenter_nodeclaims_disrupted_total broken down by reason. If you see a spike in drift-reason disruptions, a new AMI was published and your fleet is being updated. If you see frequent consolidation disruptions, your workloads might be scaling up and down rapidly — consider increasing consolidateAfter.

5. Set Resource Limits as a Safety Net

Always set limits on your NodePools. Without limits, a misconfigured deployment (or a scaling bug) could cause Karpenter to provision hundreds of nodes:

limits:
  cpu: "200"       # No more than 200 vCPUs total in this pool
  memory: 800Gi    # No more than 800 GiB total memory

When the limit is reached, new pending pods won't trigger provisioning — they'll stay Pending. Set alerts on pods in Pending state so you can decide whether to increase limits or investigate the source of demand.

What the Numbers Look Like

Here's a realistic before-and-after for a team migrating from ASG-based node management to Karpenter on a mid-size EKS cluster:

Metric Before (ASG-Based) After (Karpenter)
Node provisioning time 3–5 minutes 60 seconds
Instance type selection Fixed per node group Dynamic per provisioning decision
Average node utilisation 35–45% 65–80% (active consolidation)
Monthly compute spend (20 services) $14,000 $7,500
AMI patching time 2–4 hours (rolling update pipeline) Automatic (drift detection)
Spot interruption recovery 3–5 min (NTH + CAS) ~60 seconds (pre-provisioned replacement)
Infrastructure to manage 5 ASGs, 3 launch templates, NTH DaemonSet 2 NodePools, 1 EC2NodeClass
Annual savings — ~$78,000

The savings come from three sources: tighter bin-packing (right-sized instances instead of oversized fixed types), active consolidation (fewer underused nodes), and faster Spot recovery (less time paying for replacement capacity). The operational savings — eliminating ASG management, launch template maintenance, NTH upgrades, and AMI patching pipelines — are harder to quantify but just as real.

A Note on EKS Auto Mode

Since December 2024, EKS Auto Mode runs Karpenter as a fully managed, off-cluster component. AWS manages the Karpenter installation, upgrades, and operational health as part of the EKS control plane. You still define NodePool and EC2NodeClass resources the same way — the declarative API doesn't change.

If you're setting up a new cluster and want Karpenter with zero operational overhead, EKS Auto Mode is the simplest path. The tradeoff is less visibility into Karpenter's internals and less control over its version and configuration. For most teams, the managed defaults are good enough.

Key Takeaways

  1. Two CRDs replace your entire node scaling infrastructure. NodePool defines what Karpenter is allowed to provision. EC2NodeClass defines the AWS-specific configuration. No ASGs, no launch templates, no managed node groups.
  2. Nodes appear in 60 seconds, not 3–5 minutes. Karpenter calls the EC2 Fleet API directly — no ASG intermediary. It batches pending pods and selects the cheapest instance type that fits them all.
  3. Drift detection keeps your fleet current automatically. When a new AMI is published or you change security groups, Karpenter detects the divergence and gracefully replaces affected nodes — no manual patching pipeline needed.
  4. Consolidation actively optimises cost. Karpenter doesn't just remove empty nodes — it replaces underused nodes with smaller, cheaper ones. Three 25%-utilised m5.xlarge instances become one right-sized m5.large.
  5. Expiration guarantees node freshness. Set expireAfter: 720h and every node gets replaced at least once a month — regardless of drift. Security patching becomes a non-event.
  6. Spot interruptions are handled natively. No DaemonSet needed. Karpenter detects interruption events via SQS, pre-provisions replacement nodes before draining, and responds to rebalance recommendations proactively.
  7. Disruption budgets give you control. Limit how many nodes are disrupted at once, schedule maintenance windows, and annotate critical pods with do-not-disrupt to prevent unwanted moves.
  8. Separate NodePools for different workload classes. On-demand for critical services, mixed for general purpose, Spot-only for batch. Use weights to control priority.
