Talk to Your Cluster: Kubernetes MCP Server in Practice

Richie Richard Rajkumar · April 2026 · 10 min read

If you've spent any time operating Kubernetes clusters, you know the drill: kubectl get pods -n production | grep -v Running, then kubectl describe pod, then kubectl logs, then kubectl get events --sort-by=.lastTimestamp — and you're five commands deep before you even understand what went wrong. What if you could just ask?

The Kubernetes MCP Server is an open-source tool that connects AI assistants (Claude, VS Code Copilot, Cursor) directly to the Kubernetes API via the Model Context Protocol. It's written in Go, talks to the K8s API natively — no kubectl wrapper — and turns natural language into real cluster operations. I've been running it against staging and production for three weeks. Here's what actually works, what surprised me, and how to set it up without accidentally deleting your prod namespace.

What Is MCP and Why Does It Matter for Kubernetes?

The Model Context Protocol (MCP) is an open standard that lets AI models call external tools in a structured way. Instead of the AI generating kubectl commands for you to copy-paste, MCP lets the AI execute operations directly against your cluster through a well-defined interface — with guardrails you control.

The kubernetes-mcp-server implements this protocol specifically for Kubernetes. It exposes your cluster's state and operations as MCP tools that any compatible AI assistant can invoke. Think of it as giving your AI assistant a kubeconfig and the knowledge to use it intelligently.

Aspect Traditional kubectl Workflow With MCP Server
Investigating a crash 4-6 sequential commands, manual correlation "Why is the payment pod crashing?" — one question
Cross-namespace audit Scripted loop across namespaces, parse JSON "List all pods with more than 3 restarts across all namespaces"
Helm release status helm list + helm status + kubectl get "What version of the ingress chart is running and is it healthy?"
Context switching kubectl config use-context per cluster "Compare pod counts between staging and production"
Learning curve Memorise flags, resource types, JSONPath Describe what you want in plain English
Key point: This is not a kubectl wrapper that parses text output. It's a native Go binary that communicates directly with the Kubernetes API server using client-go. The difference matters — you get structured data, proper error handling, and no parsing fragility.

Setting It Up in 5 Minutes

The server runs as a local process that your AI assistant connects to via stdio. Prerequisites: a working kubeconfig and cluster access.

Step 1 — Verify Cluster Access

# Make sure you can talk to your cluster
$ kubectl get nodes
NAME           STATUS   ROLES           AGE   VERSION
node-pool-01   Ready    <none>          45d   v1.30.2
node-pool-02   Ready    <none>          45d   v1.30.2
node-pool-03   Ready    <none>          45d   v1.30.2

Step 2 — Install the Server

Pick your preferred method. npm is the fastest for local use:

# Option 1 — npm (no build required)
$ npm install -g kubernetes-mcp-server

# Option 2 — Python
$ pip install kubernetes-mcp-server

# Option 3 — Download the binary directly
$ curl -L https://github.com/containers/kubernetes-mcp-server/releases/latest/download/kubernetes-mcp-server-$(uname -s)-$(uname -m) -o /usr/local/bin/kubernetes-mcp-server
$ chmod +x /usr/local/bin/kubernetes-mcp-server

# Option 4 — Helm (for running inside the cluster)
$ helm install k8s-mcp oci://ghcr.io/containers/kubernetes-mcp-server/helm/kubernetes-mcp-server

Step 3 — Configure Your AI Client

Add the MCP server to your client's configuration. Here's what it looks like for Claude Desktop, VS Code, and Cursor:

// Claude Desktop — ~/.claude/claude_desktop_config.json
{
  "mcpServers": {
    "kubernetes": {
      "command": "kubernetes-mcp-server",
      "args": ["--read-only"]
    }
  }
}

// VS Code — .vscode/mcp.json
{
  "servers": {
    "kubernetes": {
      "command": "kubernetes-mcp-server",
      "args": ["--read-only"]
    }
  }
}
Start with --read-only. Seriously. Get comfortable with the query capabilities first. You can remove this flag later when you trust the workflow. I ran read-only for the first two weeks.

Real-World Examples: What It Actually Looks Like

Here are real conversations I've had with Claude while the MCP server was connected. These aren't hypothetical — they're from actual incident triage and daily operations.

Example 1 — Finding Unhealthy Pods Across All Namespaces

You:
Show me all pods that aren't Running or Completed across every namespace.
Claude:
I found 3 pods in non-healthy states:

payments/payment-processor-7d4f8b-xk2mn — CrashLoopBackOff (12 restarts, last restart 3m ago)
monitoring/prometheus-node-exporter-9p2ql — Pending (insufficient CPU, requested 500m but node has 120m available)
batch/daily-report-28847201-wt5kz — Error (exit code 137, OOMKilled)

The payment-processor has been crash-looping since 14:32 UTC. Want me to pull its logs?

Without the MCP server, this would have been:

$ kubectl get pods -A --field-selector=status.phase!=Running,status.phase!=Succeeded
$ kubectl describe pod payment-processor-7d4f8b-xk2mn -n payments
$ kubectl get events -n payments --sort-by=.lastTimestamp
# ... and so on for each pod

Example 2 — Debugging an OOMKilled Container

You:
The daily-report job is OOMKilling. What are its memory limits and what was the actual usage?
Claude:
Here's what I found for batch/daily-report:

Container spec:
Memory request: 256Mi | Memory limit: 512Mi
CPU request: 250m | CPU limit: 500m

Last termination:
Reason: OOMKilled | Exit code: 137
Terminated at: 2026-04-08T06:00:47Z

The container was killed at exactly the 512Mi limit. Looking at the pod's previous runs, 3 of the last 5 executions were OOMKilled. The successful runs used ~480Mi peak — this container is right at the edge. I'd recommend bumping the limit to at least 768Mi and the request to 512Mi.

Example 3 — Helm Release Inspection

You:
List all Helm releases in the cluster. Flag any that are in a failed state or have pending upgrades.
Claude:
Found 14 Helm releases across 6 namespaces:

Failed (1):
monitoring/kube-prometheus-stack — v62.3.0 → v63.1.0 upgrade failed (last attempted 2h ago). The release is still running v62.3.0. Error: "cannot patch CRD servicemonitors.monitoring.coreos.com"

Pending (1):
ingress/nginx-ingress — upgrade to v4.11.3 is pending-upgrade since 09:14 UTC

Deployed (12):
All healthy. Oldest release: cert-manager v1.16.1, deployed 62 days ago.

Example 4 — Multi-Cluster Comparison

If your kubeconfig has multiple contexts, the MCP server can operate across them:

Compare the api-gateway deployment between staging and production. Are they running the same image version?
Claude:
staging (context: apps-sandbox-eu-central-1):

Example 5 — Scaling a Deployment (Write Mode)

With --read-only removed, the MCP server can perform mutations:

You:
Scale the api-gateway deployment in production to 8 replicas. We're expecting a traffic spike from the product launch.
Claude:
Scaled production/api-gateway from 5 → 8 replicas.

Current rollout status: 8/8 replicas available (all new pods scheduled and running within 45 seconds). No resource pressure detected on the node pool — the cluster has 12 allocatable CPU cores and 28Gi memory remaining.

Production Guardrails: Don't Skip This Section

Connecting an AI assistant to your production cluster without guardrails is asking for trouble. The MCP server has several safety mechanisms — use all of them.

Read-Only Mode

# The simplest and most important guardrail
$ kubernetes-mcp-server --read-only

# All queries work. All mutations are blocked.
# Perfect for monitoring, triage, and auditing.

Disable Specific Destructive Operations

# Allow most writes but block deletions and exec
$ kubernetes-mcp-server \
    --disable-delete \
    --disable-exec

Restrict Access to Sensitive Resources

Use a TOML config file to block access to specific resource types entirely:

# mcp-config.toml
[restrictions]
blocked_resources = [
  "secrets",
  "serviceaccounts",
  "clusterroles",
  "clusterrolebindings"
]

blocked_namespaces = [
  "kube-system",
  "kube-public"
]
$ kubernetes-mcp-server --config mcp-config.toml

RBAC-Scoped Service Account (In-Cluster Deployment)

When running the MCP server inside the cluster via Helm, bind it to a service account with minimal permissions:

apiVersion: rbac.authorization.k8s.io/v1
kind: ClusterRole
metadata:
  name: mcp-readonly
rules:
  - apiGroups: [""]
    resources: ["pods", "services", "events", "nodes", "configmaps", "namespaces"]
    verbs: ["get", "list", "watch"]
  - apiGroups: ["apps"]
    resources: ["deployments", "replicasets", "statefulsets", "daemonsets"]
    verbs: ["get", "list", "watch"]
  - apiGroups: ["batch"]
    resources: ["jobs", "cronjobs"]
    verbs: ["get", "list", "watch"]
Defense in depth: Use --read-only mode and a restricted RBAC role. If a bug in the MCP server bypasses the read-only flag, the service account's RBAC permissions are the second barrier. Neither layer alone is sufficient for production.

Three Workflow Patterns That Changed How I Operate

Pattern 1 — Morning Cluster Health Check

Instead of a checklist of kubectl commands, I start the day with one prompt:

You:
Give me a health summary of the production cluster. Focus on: pods not running, recent events with type Warning in the last 2 hours, nodes with resource pressure, and any deployments not at desired replica count.
Claude:
Cluster: apps-production-eu-central-1 (3 nodes, 47 pods)

Pods: All 47 pods Running or Completed. No crash loops. Highest restart count: monitoring/alertmanager-0 at 2 restarts (last restart 18h ago, within normal range).

Warning events (last 2h): 3 events
- FailedScheduling on batch/etl-daily-28847500: "0/3 nodes available: insufficient memory" (resolved, job completed on retry)
- Unhealthy readiness probe on api/search-service-6b8c4-pm9kz (1 occurrence, pod recovered)
- ImagePullBackOff on staging/feature-preview-abc123 (image tag :latest not found, likely a stale deployment)

Nodes: No resource pressure. node-pool-01 is at 72% memory (highest), healthy.

Replica drift: None. All deployments at desired count.

That replaces six separate kubectl + jq pipelines I used to run manually every morning.

Pattern 2 — Incident Triage Runbook

When a PagerDuty alert fires for a failing pod, I paste the alert context and let the MCP server run the investigation:

You:
PagerDuty alert: "pod payment-processor in namespace payments has been CrashLoopBackOff for 10 minutes". Investigate: get the last 100 lines of logs, describe the pod, check recent events, and check if the ConfigMap or Secret it mounts has been modified recently.
Claude:
Root cause identified: The pod is crash-looping due to an invalid database connection string.

Logs (last 100 lines):
FATAL: password authentication failed for user "payment_svc" — repeated 12 times since 14:32 UTC.

ConfigMap payment-processor-config:
Last modified: 14:29 UTC (3 minutes before the crash loop started). The DB_HOST value was changed from postgres-primary.db.svc to postgres-replica.db.svc. The replica may require different credentials.

Recommendation: Either revert the ConfigMap to point back to postgres-primary.db.svc, or update the Secret payment-db-creds with the credentials that work for the replica endpoint.

Pattern 3 — Pre-Deploy Validation

You:
I'm about to deploy v2.14.0 of the api-gateway to production. Before I do, check: current rollout status, available node resources for 3 additional pods, and whether the image exists in our registry.
Claude:
Pre-deploy check for api-gateway v2.14.0:

Current version: v2.13.1, 5/5 replicas healthy, last rollout completed 3 days ago — clean baseline.

Node capacity for +3 pods: The api-gateway requests 250m CPU and 512Mi per pod. The cluster has 4200m CPU and 12Gi memory allocatable — room for 16 additional pods. No concern here.

Image: ghcr.io/acme/api-gateway:v2.14.0 exists (pushed 2 hours ago, 142MB). Ready to deploy.

Limitations: Where It Falls Short

It's worth being honest about where this tool doesn't (yet) deliver:

  • No real-time streaming. You can't tail -f logs through the MCP server. It fetches a snapshot. For live log tailing during incidents, you still need kubectl logs -f or a log aggregator like Loki.
  • CRD awareness is limited. It can list and describe custom resources, but it doesn't understand the semantics of your ArgoCD Applications or Istio VirtualServices the way it understands core Kubernetes resources.
  • No port-forwarding. You can't ask it to set up a port-forward for debugging. That's inherently interactive and long-lived — not a good fit for the request/response MCP model.
  • Context window limits. If you ask for logs from a pod that outputs 50,000 lines, the AI's context window becomes the bottleneck, not the MCP server. Be specific: "last 100 lines" or "lines containing ERROR".
  • Audit trail. The MCP server doesn't log what operations were performed. For production write access, you need your own audit logging — Kubernetes audit logs cover the API calls, but the MCP server itself is stateless.

Who Benefits Most

Role Primary Use Case Mode
SREs / On-Call Incident triage, event correlation, multi-cluster status checks Read-only
DevOps Engineers Deployment validation, Helm operations, resource auditing Read-only or selective write
Platform Engineers Cluster standardisation audits, RBAC reviews, capacity planning Read-only
Developers Debug failing pods in dev/staging, understand resource requests Read-write (non-prod only)

Getting Started Checklist

  1. Install the binary — npm install -g kubernetes-mcp-server is the fastest path
  2. Start in read-only mode — add --read-only to your MCP config and explore for a week before considering write access
  3. Scope your kubeconfig — only include the contexts you actually want the AI to access. Don't point it at every cluster in your org
  4. Block sensitive resources — use a TOML config to block Secrets, ServiceAccounts, and RBAC resources from queries
  5. Add RBAC as a second layer — even with --read-only, use a dedicated service account with minimal ClusterRole bindings
  6. Test against staging first — run it against a non-production cluster for at least a few days. Get a feel for what it surfaces and how it interprets ambiguous requests
  7. Graduate to selective writes — when you're confident, enable write mode with --disable-delete --disable-exec so you can scale and patch but can't destroy
My take: The Kubernetes MCP Server isn't going to replace kubectl — and it shouldn't. What it does is eliminate the tedious multi-step investigation workflows that eat up time during incidents and daily operations. The value isn't in any single query — it's in the compounding effect of being able to ask follow-up questions that automatically correlate across pods, events, logs, and node state. The productivity difference is most visible during on-call: instead of context-switching between terminal tabs, you stay in one conversation and let the AI assemble the picture. Start read-only, scope it tight, and let it earn your trust incrementally.
